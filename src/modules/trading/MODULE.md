# trading (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns (tables/routes/providers), or its dependencies on other modules.

## Purpose

The **Casino-Mode trade screen** (PRD-0008 §8, D7/D16/D17/D18). A punter picks a market (from the lobby / search), taps a bet amount and a multiplier, then bets **UP** or **DOWN** — which opens a real leveraged Hyperliquid perp via a market IOC. Same real trading as the perps client, re-skinned as a casino: fewer numbers, bigger buttons, casino vocabulary, and **notional is never shown**.

The screen talks to the active **Venue** through capability ports only (`Trader`, `MarketDataReader`, `CandlesReader`, plus the **`ownAccount`** group — `portfolio` / `perpsPositionsSnapshot` keyed to the **Acting Address**, read for available balance and the open bet so they show the User's own account even while **Spectating**, ADR-0038). It **never imports a venue module** directly (lint-enforced). Order placement + the hidden HL agent setup, and the open-bet Cash Out, both reach the venue via the `Trader` port.

## Public surface

> **PRD-0008 casino re-skin — status.** The AI/Minara suggestion surface, the Pro-mode split, and the orderbook / recent-trades / depth / funding / `TickerStats` surfaces are **deleted** (§9). The **Casino-Mode trade screen is now built** — `TradingPage` is the chart + bet-amount chips + multiplier + UP/DOWN commit + confirm sheet + live-bet row described below. The legacy `OrderEntry` / `LeverageMargin` component trees remain in the folder as internal building blocks (they are the source of the reused `ORDER_CLOID_PREFIX` constant and the `use-live-mark` ticker hook) but `OrderEntry` is **no longer mounted** by any page.

- `TradingPage` — the route-addressable Casino-Mode page (mounted by `app/router.tsx`'s `/trade/:symbol`, wrapped in `SelectedMarketProvider → OrderIntentProvider → LeverageMarginProvider`). Renders the market header (● ticker · price · 24h change), the lightweight-charts price chart (`LazyChart`), the open **live-bet row** (when a position on this market is open), the **YOUR BET** amount chips, the **MULTIPLIER** slider, the **UP/DOWN** commit buttons, and the bottom-sheet **confirm bet** flow. The 3-column app shell, the LIVE WINS ticker, and the chat column are owned by `app/AppShell`; `TradingPage` is the centre trade column only. Its brain is `useCasinoTrade` (`hooks/`). Single responsive column — no separate mobile tree.
- `HotMarketsTicker` — the global-header "hot markets" running row (a seamless CSS marquee of the selected venue's top markets by 24h volume). Mounted by `app/AppShell`. **Route-independent by design:** it reads the venue market universe via the `marketData` capability and selects via the **URL contract** (`useNavigate` → `/trade?market=hl:<symbol>`), **never** consuming `SelectedMarketProvider`. "hot" = top-N by 24h volume via `pickHotMarkets`. Smart hook + dumb sub-components live in `components/hot-markets-ticker/`.
- `MobileTradeDock` / `MobileBottomNav` — the mobile footer nav composite (retained module exports consumed by the casino app shell / other pages). Not mounted by the Casino-Mode `TradingPage`.
- `MarketSelectionWindow` — the market list, demoted by D15 from primary nav to the app shell's **search overlay**. Internal tree + smart hook; re-exported for the shell to mount.
- `SelectedMarketProvider` — context for the currently selected market. `useSelectedMarketContext()` is `{ selectedMarket: MarketSymbol; setSelectedMarket; market: Market }`: the resolved domain `Market` (carrying `hlCoin` / `marketType` / `stepSize` / `maxLeverage`) is resolved once here from the venue's `listMarkets()`/`subscribeMarkets`. Only the symbol string is persisted to URL/storage.
- `LeverageMarginProvider` — owns the order form's leverage + margin-mode state for the selected market as **shared** state (`useLeverageMargin()`). Casino Mode reads `leverage` / `maxLeverage` for the multiplier and applies changes via `applyLeverage` (which signs the venue `setLeverage`). Seeded from the Acting `perpsPositionsSnapshot`, falling back to 1×/cross.
- `OrderIntentProvider` — the order-intent bus (retained; the former suggestion publisher was deleted). Not consumed by Casino Mode.
- `FavoritesProvider` — persisted favorites set (`useFavorites()` / `useFavoritesOptional()`). Mounted **app-wide** in `AccountSessionRoot`: the market list, the search overlay, and the lobby's `/?view=favorites` grid all share one store.
- `RecentMarketsProvider` — the markets most recently opened (`useRecentMarkets()` / `useRecentMarketsOptional()`), newest first. Mounted app-wide next to `FavoritesProvider`; feeds the lobby's `/?view=recent` grid.
- `TradeDockProviders` — the context stack (`SelectedMarketProvider → OrderIntentProvider → LeverageMarginProvider`) for pages that host the mobile dock without the `/trade` nesting.
- `MarketSymbol`, `DEFAULT_SELECTED_MARKET` — the cross-module market-symbol type + default.

## Owns

### Casino-Mode trade screen (PRD-0008 §8)

- **`hooks/use-casino-trade`** — the screen brain. Owns the bet amount (margin, seeded to the first preset `$10`), the multiplier (via `useLeverageMargin`), and the confirm-sheet direction/open state. Reads the live mark (`use-live-mark`), the Acting `portfolio` snapshot (`'perps'` scope) for available balance, and the `Trader` port for the confirm-sheet liquidation preview. Derives `size` via the D18 `marginToSize` conversion, `canBet`, and the `PendingBet` view (direction, amount, multiplier, **liquidation sentence**, resolved CTA, and the built `OrderDraft`). `MAX` sets the bet amount to the full available margin (D18). `confirmPrimary` dispatches on the resolved CTA: `connect` → `useAuth().openConnectModal`, `add-cash` → `useManageFunds().open('deposit')`, `place-bet` → `usePlaceBet.place`.
- **`hooks/use-place-bet`** — the place-bet action (D6/D17). Validates the draft through `Trader.validateDraft`, attaches a fresh cloid, and submits `placeOrder` (market IOC). When the venue still needs onboarding — the hidden HL **agent approve-and-register** — it runs `useVenueOnboarding().runAll(defaultOnboardingValues(steps))` FIRST behind a `setting-up` phase ("Setting up your table…", no modal, no "agent wallet" copy), then places. **REUSES** the shared onboarding provider (which drives `hyperliquid/`'s `agent-wallet-provider`); that provider mints a **fresh keypair per approval** with a stable native-keyed name (HL anti-replay, ADR-0077), so the silent path is safe to re-run.
- **`hooks/use-live-bet`** — the live-bet surface. Subscribes to the User's own `perpsPositionsSnapshot` (Acting-keyed) and, when a position on the current market is open, projects `{ direction, leverage, profitUsd, isWinning, liquidationSentence }` and a `cashOut` that market-closes the **full** position (reduce-only opposite-side market order via `Trader`).
- **`hooks/casino-trade.utils`** (unit-tested) — the pure core: `marginToSize(betAmount, leverage, markPrice, szDecimals)` (**D18** — `size = margin × leverage / markPrice`, rounded to the market lot precision recovered from metadata via `specFromMarket`; the `$500` notional is never surfaced); `formatLiquidationSentence(direction, price, market)` (**D16** — "You lose this bet if BTC drops below $94,102" / "…rises above…", always shown, never a labelled number); `resolveConfirmCta` (**D6** gating — `connect` → `add-cash` → `setting-up` → `placing` → `place-bet`); `buildBetDraft` (coin-unit market IOC draft — no limit/TIF/reduce-only surface, D17); `directionToSide` / `positionSideToDirection`; `defaultOnboardingValues` (harvests step-input defaults so the silent onboarding runs form-less).
- **`components/casino-trade/`** — the dumb component set (`MarketHeader`, `BetAmountChips`, `MultiplierControl`, `DirectionButtons`, `ConfirmBetSheet`, `LiveBetRow`), each < 200 lines, props typed in `casino-trade.types.ts`, styled by the shared `casino-trade.module.css`. `DirectionButtons` uses `PixelButton` `directionUp` (`--win`) / `directionDown` (`--loss`), min-height 64px; `ConfirmBetSheet`'s **PLACE BET** is the only `--accent-action` **magenta** commit (D16 sheet). `MultiplierControl` tracks a local slider draft and commits (`applyLeverage`, which signs) on release only. `MarketHeader` renders the market price via `formatUsd` and the 24h change (a signed fraction) coloured `--win`/`--loss`.

### Retained infrastructure

- `SelectedMarketProvider` — the selected-market state + the **URL contract** (`?market=hl:<symbol>` source of truth, `localStorage` fallback; the `/trade/:symbol` path seeds it). Pure helpers `parseMarketParam` / `formatMarketParam` / `buildTradeMarketHref`. Single URL-preferring reconciler effect (no two fighting writers).
- `LeverageMarginProvider` + `use-leverage-margin-state` — shared leverage/margin-mode state; `applyLeverage`/`applyMarginMode` sign the venue controllers and reflect the applied value locally (the stream reconciles). Seeded from the Acting `perpsPositionsSnapshot` (ADR-0038 D-1). The inline `LeverageMargin` section + `OrderEntry` component trees remain as internal building blocks but are **not mounted** by the Casino-Mode page.
- `FavoritesProvider` — versioned `localStorage` favorites (`favorites-store.ts`, key `perps-dex-favorites`), `Set<MarketSymbol>`, `reconcileFavorites()` drops delisted symbols.
- `RecentMarketsProvider` — versioned `localStorage` visit list (`recent-markets-store.ts`, key `perps-dex-recent-markets`, cap `RECENT_MARKETS_LIMIT` = 12, newest first). The pure `recordRecentMarket()` prepends → dedups → caps, so re-visiting a market **promotes** it rather than duplicating it. The key is **global, not per-`privyId`** (matching favorites): it is a browsing convenience with no PII, and a logged-out user browsing `/trade/:symbol` must still get a Recent list. A corrupt or stale-shape stored value degrades to an empty list — only a throwing `localStorage` is an `err`.
  - **`use-selected-market-provider` is the only writer.** Not `TradeRoute`: this provider is the one place that sees *every* visit (deep links, lobby cards, the ticker's `?market=`, the search overlay's in-page `setSelectedMarket`, back/forward), and it is the one that resolves the symbol against the live universe — so an unknown or delisted `/trade/FOO` never pollutes the list. Its effect re-runs on every venue snapshot; the `recentSymbols[0] === symbol` head-guard inside `recordMarketVisit` is what makes those re-runs free (no write, no re-render). **Do not drop that guard.**
  - There is deliberately **no** `reconcileRecent`. The lobby intersects the stored list with the live universe at read time, so a delisting drops out on its own and ages past the 12-cap.
- **Module-level types/constants** (`trading.types.ts` / `trading.constants.ts` / `trading.utils.ts`) — `MarketCategory` / `MarketCategoryTab` / `MARKET_CATEGORY_TABS`, `MIN_MARKET_VOLUME_USD` (`500_000`), the per-class symbol sets, `getMarketCategory`, and `filterByMinVolume` (the canonical liquidity floor), consumed by the Market Selection window.
- Internal cross-component hooks `use-adapter-stream` (generic stream-reducer, private).
- `components/market-selection-window/` — the MarketSelectionWindow tree (asset-class tabs, liquidity-floored via `filterByMinVolume`), mounted by the app shell's search overlay.

## Gotchas

- **`trading/` must never import a venue module** (`mock-venue/`, `hyperliquid/`, …). It depends on capability ports + the shared venue-agnostic providers (`venue-onboarding-provider`, `manage-funds-provider`) only. Lint-enforced.
- **D18 — the chip is MARGIN, not notional.** `marginToSize` converts it; the draft is submitted `sizeUnit: 'coin'` with that computed size, so the placed size is exactly what the tested util produced (no re-conversion drift). Notional (`size × mark`) is never rendered anywhere.
- **D6 — never render "agent wallet" / "signing key" / "nonce" / "gas".** The first-bet onboarding runs silently behind the "Setting up your table…" CTA via `useVenueOnboarding().runAll`. Do not reach into `hyperliquid/`'s `agent-wallet-provider` from `trading/` — it is a lint-forbidden venue import; the shared onboarding provider is the seam.
- **The confirm `Sheet` opens non-modally** (`show()`, not `showModal()`) so it sits at `--z-sheet` below Privy's body-level wallet modal — the connect flow must overlay the sheet. jsdom does not implement `<dialog>.show()`; component tests that render the real `Sheet` must stub it.
- A venue switch hard-remounts the trading subtree (ADR-0001) — keep in-page state stateless w.r.t. venue identity, or accept the remount.
