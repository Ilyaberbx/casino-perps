# account (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns (tables/routes/providers), or its dependencies on other modules.

## Purpose

Owns the **User** identity story on the client: wraps Privy as the only identity provider, surfaces it as `useAuth()`, runs the **Onboarding** flow against our backend, and exposes the typed `apiClient` that every other module uses to talk to the server. See `CONTEXT.md` for `User`, `Privy ID`, `Wallet`, `Primary Wallet`, `Onboarding`.

## Public surface

- `AuthProvider` — wraps `PrivyProvider` (configured `loginMethods: ['email']` — email-OTP is the **sole** onboarding path per PRD-0006 Slice 03 / ADR-0058; `embeddedWallets.ethereum.createOnLogin: 'users-without-wallets'` keeps the Native embedded wallet; **`mfa: { noPromptOnMfaRequired: false }`** keeps Privy's own MFA *verification* modal ON, so the MFA-gated private-key export (ADR-0076 D-5) can force MFA before releasing a key — this `@privy-io/react-auth` v3 config is the equivalent of the older `enableMfaVerificationUIs: true` flag, which does not exist in the installed SDK) and mounts the `OnboardingStepper` once inside its subtree (via `AuthValueProvider`). The rest of the app never imports from `@privy-io/react-auth` directly. **Fallback embedded-wallet provisioning:** `AuthBridge` runs `useEnsureEmbeddedWallet` (`providers/auth-provider/use-ensure-embedded-wallet.ts`) — once Privy has settled, if an authenticated user has **no** wallet (`user.wallet == null` and no `privy`/`privy-v2` embedded entry in `linkedAccounts`) it explicitly calls Privy `useCreateWallet().createWallet()` (once per user). This closes the dead-end where `createOnLogin` is suppressed upstream (e.g. embedded wallets disabled for the app in the Privy dashboard) and the user is stranded with no `walletAddress` → onboarding never starts; a rejected creation surfaces a `embedded-wallet-create-failed` error toast (the benign `embedded_wallet_already_exists` race is ignored) so a dashboard misconfig is visible, not silent.
- `useAuth` / `AuthContext` / `AuthState` — auth read API. Also surfaces `hasMfa: boolean` — the **only** client-visible 2FA signal, derived in `AuthBridge` from `user.mfaMethods?.includes('totp')` (Privy surfaces an enrolled authenticator-app factor as the string `'totp'` in `user.mfaMethods` — there is no server MFA flag and no server-side WebAuthn). It is `false` until Privy resolves the user and flips `true` after `enrollMfa` enrolls a TOTP factor and the Privy user re-resolves. The Account Modal's binary 2FA state reads this. Also surfaces `enrollMfa(): ResultAsync<void, AuthError>` — wraps Privy's `useMfaEnrollment().showMfaEnrollmentModal()` (via `ResultAsync.fromPromise` + `coerceToAuthError`). Optional and skippable; called by the onboarding stepper's MFA step and the Account Modal's 2FA section. Surfaces `walletAddress: string | null` — **Privy's canonical linked wallet (`user.wallet.address`), NOT `useWallets()[0]`**. With an external wallet (e.g. MetaMask) `useWallets()` reflects the currently-selected injected account, which is routinely a different address than the one Privy linked at sign-up; the server verifies onboard against Privy's `linkedAccounts`, so the Primary Wallet must be the linked one. Also surfaces `primaryWalletAddress: WalletAddress | null` — the same address parsed through `parseWalletAddress` from `shared/domain` and lower-cased at the boundary so consumers receive the branded type without re-parsing. Includes `openConnectModal()` / `closeConnectModal()` / `isConnectModalOpen` to drive the onboarding stepper from anywhere. `loginWithWallet` is unchanged and still on `AuthState`. Email-OTP itself is driven inside the stepper via `useLoginWithEmail()` (`sendCode` / `loginWithCode` / `state`), not through an `AuthState` handler. As of Slice 06, `AuthState` also surfaces `linkWallet(): ResultAsync<string, AuthError>` — opens Privy's link-wallet modal and resolves with the linked **external** wallet address (built in `AuthBridge` via the `useLinkExternalWallet` hook over `useLinkAccount`; the Wallets section feeds that address to `POST /api/account/wallets/import`) — and `externalWallets: ReadonlyArray<{ address: string; walletClientType: string }>`, the live external (non-`privy`) wallets with their connector `walletClientType`, used for best-effort connector-brand icons on imported Wallet rows (empty for embedded-only users). **As of the self-custodial-agent-wallet-export slice (ADR-0076), `AuthState` also surfaces:** `exportableAddresses: ReadonlyArray<string>` — the lower-cased addresses of every **Privy-managed** wallet (`walletClientType === 'privy'`: the embedded Native wallet, raw-key imports, and a now-user-owned Agent Wallet), i.e. the only wallets Privy holds key material for and that the owner can export (D-5/D-6). It is the runtime guard for the Wallets section's **Export private key** affordance (`Wallet.source` is the provenance source of truth; `exportableAddresses` is the live cross-check). It **includes** the user-owned Agent Wallet on purpose (so its export affordance lights up), but is **not** a trading-selection source — selectable masters come only from `me.wallets` (G-6, never the Agent Wallet), and `externalWallets` already excludes the agent (it filters `!== 'privy'`), so the user-owned agent can never leak into the trading-selectable master list. `exportWallet(address: string): Promise<void>` — owner-only, MFA-gated key export; wraps Privy's `useExportWallet().exportWallet({ address })` (Privy loads the key on an isolated iframe and forces MFA verification; the app never sees it). The MFA gate (enrol-if-absent → export) is orchestrated by `hooks/use-wallet-export.ts`, not here. `importPrivateKey(hex: string): Promise<{ address: string }>` — wraps Privy's `useImportWallet().importWallet({ privateKey })`; returns the new wallet address lower-cased (the caller then POSTs it to `/api/account/wallets/import` with `source: 'imported'`). The raw key is handed straight to Privy and never logged. **As of the client-created Agent Wallet slice (ADR-0078), `AuthState` also surfaces the three Agent Wallet seams (the only place Privy's `useCreateWallet` / `useSigners` are touched):** `createAgentWallet(): Promise<{ address: string; walletId: string }>` — wraps `useCreateWallet().createWallet({ createAdditional: true })` to mint a **second, user-owned** embedded wallet distinct from the Native one (the `use-ensure-embedded-wallet` path); returns the lower-cased address + Privy server `walletId` the caller registers with the server. `attachAgentSigner({ address, appSignerId, policyId }): Promise<boolean>` — wraps `useSigners().addSigners(...)` to attach the app as a scoped additional signer; resolves `true` on success, `false` on owner decline (a `cancelled` AuthError), rethrows a genuine failure. `removeAgentSigner({ address, appSignerId }): Promise<boolean>` — wraps `useSigners().removeSigners({ address })` (removes all signers on the wallet) for delegation revoke; mirrors `attachAgentSigner`'s throw handling — resolves `true` on success OR when a benign Privy throw is swallowed (best-effort, e.g. the signer is already gone) so the authoritative server row-revoke still runs, and `false` only on owner decline (a `cancelled` AuthError); the swallowed cause is logged via `describeAuthError`. All three are consumed by `agent-balance/` only through `useAuth()` — that module never imports the Privy SDK (import-boundary asserted by test).
- `AccountAvatarTrigger` — UI. The header account entry point (PRD-0006 UI-2, Slice 04). Renders a square pixel-framed **web3-avatar** (seeded from the Native/embedded wallet address via `Web3Avatar`) + the user's handle; clicking calls `useAccountModal().open()`. While the onboarding FSM is resolving (the trigger only mounts once authenticated, so non-`ready` ⇒ authenticated-but-loading) it renders a **loading skeleton** (`ValueSkeleton` avatar + handle bars) sized to the trigger's exact footprint, so the button reveals in place instead of popping in; it renders `null` only when a `ready` `Me` has no Native wallet. The unauthenticated Connect entry point lives in `app/`'s `AppShell`. **Retired the old `AccountMenu` dropdown** in this slice: the header avatar now opens the Account Modal directly (and Log out moved into the modal's Profile footer).
- `QuickWalletSwitcher` — UI. The header quick-action wallets switcher, rendered in `AppShell` beside the `VenueSwitcher` (desktop header + the mobile `mobileHeaderControls`). Shows the **Selected Wallet** (`Web3Avatar` + `Native`/truncated-address label) and opens a portaled `Popover` menu to switch the Selected Wallet or **Import wallet** — a condensed shortcut to the Account Modal's Wallets section. It self-gates (renders `null` until the onboarding flow is `ready`, so it is absent when disconnected — no call-site `isConnected` branch). Collapses to an avatar-only chip under the `max-width: 560px` header-condense query (mirrors `VenueSwitcher`). **It shares all select/import behaviour with the Account Modal Wallets section** via the internal cross-component hook `hooks/use-wallet-mutations.ts` (`useWalletMutations`) — the single owner of optimistic select (`POST …/select`), import (`linkWallet` → `POST …/import`), remove (`DELETE …`), and the optimistic-selection state, all reconciled through the canonical `applyMe`. Both `use-wallets-section.ts` and `use-quick-wallet-switcher.ts` consume it, so the two surfaces never diverge from the server `is_selected`. `useWalletMutations` is module-internal (not exported through `index.ts`); only `QuickWalletSwitcher` is public.
- `AccountModal` + `AccountModalProvider` / `useAccountModal` / `AccountModalContext` — UI + provider unit (PRD-0006 UI-2/3/4, Slice 04). `AccountModalProvider` owns the `{ isOpen, open, close }` controller (mounted in `AppShell`); `AccountModal` is the shell — a centered shared `Modal` with a **left sidebar** (desktop) / **horizontal top tab strip** (mobile, via `useIsMobile`) over three sections (`Profile / 2FA / Wallets`), **default Profile**. Each section is a dumb component fed by **one** smart hook (`use-profile-section` / `use-mfa-section`; the shell's own `use-account-modal-content` owns active-section + the `isMobile` boundary). **Profile (UI-3):** square `Web3Avatar`, read-only Email + Handle (set-once, "Permanent" hint), a `CopyableAddress` on the Native address, and **Log out** (`PixelButton`, `directionDown`/danger) wired to `AuthState.logout`. **2FA (UI-4):** binary state derived from `AuthState.hasMfa` (unset → `Set up 2FA` CTA calling `enrollMfa`, error-toasts and stays — skippable-consistent with onboarding; set → "Two-factor authentication on"). **Wallets (UI-5, Slice 05):** real now. `WalletsSection` is fed by the `use-wallets-section` smart hook and renders one dumb `WalletRow` per **user** wallet (Native now; imported rows flow in unchanged in slice 06 — they are already in `me.wallets` and `toRow` labels them `Imported`) followed by a visually separated, read-only, **uncounted** `AgentWalletRow` (G-6 — the Agent Wallet is built separately and is never part of `me.wallets`, so it is structurally outside the 4-wallet cap `MAX_USER_WALLETS`). The whole non-selected row is the select affordance (`POST /api/account/wallets/:address/select` via `api/select-wallet.ts`, **optimistic** badge move, rollback + error toast on failure, 403 cross-user → "different account" toast surfaced through the `shared/services/toast` singleton — no `ToastProvider` dependency in the hook). The selected row shows a `Selected` badge + accent border. Each user row has a right overflow (⋮) `WalletRowMenu` with `Copy address`, an **`Export private key`** item for owner-exportable wallets (`isExportable`: `source` ∈ {`embedded`,`imported`} **and** the address is in `exportableAddresses`; ADR-0076 D-5) — link-only `external` wallets show a non-export note in its place — and, for imported (`external`) wallets only (Slice 06), `Remove`. Export is wired through `hooks/use-wallet-export.ts` (the MFA-gated, owner-only export hook shared by the row menu and the Agent row — ≥2 consumers). The `AgentWalletRow` carries the same Export affordance, enabled iff the (now-user-owned) Agent Wallet address is in `exportableAddresses` (its own open-state + the shared `useCopyableAddress` + the `window.confirm`-gated remove action live in `use-wallet-row-menu.ts`; the actual `DELETE` is owned by `use-wallets-section.ts`'s `onRemove`). Imported rows render their connector brand icon when resolvable (best-effort from `AuthState.externalWallets` → `account-modal.utils.ts`'s `resolveConnectorIcon`, fed to `Web3Avatar`'s `iconUrl` with graceful fallback to the deterministic gradient). The add-wallet row is a **split** (ADR-0076 D-6): a **`Connect external`** `PixelButton` (the former `Import Wallet` — `linkWallet` → `POST /api/account/wallets/import` with `source:'external'` → success toast) **and** an **`Import private key`** affordance (`components/account-modal/import-key-sheet/`: dumb `ImportKeySheet` + smart `use-import-key-sheet` over the shared **`Modal`** — a true dialog, the secret-entry surface, not the non-modal Sheet). The key sheet validates `^(0x)?[0-9a-fA-F]{64}$`, uses a masked input, **clears the key from state on close/unmount**, never logs it, and on submit calls `useAuth().importPrivateKey(hex)` → `POST /api/account/wallets/import` with `source:'imported'` → `applyMe` (list refresh). Both paths are **disabled with a `3/3 imported` hint** at the imported-wallet cap (`MAX_IMPORTED_WALLETS = 3`, counting only `source:'external'` — Native + Agent + raw-key `imported` provenance excluded from that count). `WALLET_SOURCE_LABEL` relabels `external` → **`Linked`** (link-only) and adds `imported` → **`Imported`**. Import/remove/select return the fresh `Me`, fed into the onboarding flow's canonical `applyMe` (no local override) so the list and `useSelectedWallet` stay in sync; the modal also calls `refreshMe()` on open (server re-points selection to Native on removing the Selected wallet). The Agent row is marked with the brand animated three-eyed mascot (`AiMascot animated`, `shared/components/ai-marker`) and shows the live Agent Balance (USDC) via `agent-balance`'s `useAgentBalance` (read-only — the Agent Wallet is funded from the Manage-Agent-Wallet surface, not from a per-row action). **Per-DEX balance panel (UI-5 / Slice 07):** under the wallet rows the section renders `SelectedWalletBalancesPanel` (smart sub-feature owning `use-selected-wallet-balances`) — one line per **integrated venue** (read from the active `VenueProvider`, no hardcoded Hyperliquid) — each rendering the real venue icon via the shared `VenueIcon` (`@/modules/shared/components/venue-icon` — the single venue→logo surface; a venue with no mark falls back to a first-char monogram) — showing the Selected Wallet's **Total Account Value (equity)** from the venue's `portfolio` reader; a venue the wallet isn't onboarded on (or unreadable) reads **`$0`** with a `Not onboarded — Onboard` link opening the existing `VenueOnboardingSheet` (G-10 — never an error). Display-only (no deposit deep-link). **Mount requirement:** the Wallets section reads (Slice 07) the active venue + `useVenueOnboardingSheet`, so `AccountModal` must render inside `<VenueProvider>` and `<VenueOnboardingSheetProvider>` (it does in production — `App.tsx` mounts `VenueProvider`, `AccountSessionRoot` mounts those providers above `AppShell`, which renders `<AccountModal />`). The onboarding stepper itself is **not** exported; its only consumers are `openConnectModal()` / `isConnectModalOpen` on `useAuth()`. The Hyperliquid agent + builder approvals are surfaced by the venue-onboarding sheet from `hyperliquid/`, not here (ADR-0026).
- `Web3Avatar` (`components/account-avatar/`) — UI. The shared account avatar: renders `iconUrl` when set (through `FallbackImage`, so a broken icon still degrades), else a deterministic `web3-avatar-react` `<Avatar>` gradient seeded from the Native Wallet address (lower-cased), forced **square** (no `border-radius`) inside a 2px pixel frame. Consumed by both `AccountAvatarTrigger` and the Profile section (≥2 in-module consumers). Slice 05 reuses it for per-wallet rows.
- `DisconnectedTablePlaceholder` — UI primitive. Wraps a list/table panel and, when disconnected, renders centered placeholder text (`message` prop) instead of children. Text-only — no button, no CTA. See `apps/client/.claude/rules/wallet-gate.md` for the copy convention.
- `ConnectWalletGateButton` — UI primitive. Wraps a feature submit button: renders children unchanged when connected, and renders **nothing** when disconnected (the affordance is simply absent; the Connect-Wallet entry point lives in the header — the `Connect Wallet` button shown while disconnected, which becomes the `AccountAvatarTrigger` once connected). See `apps/client/.claude/rules/wallet-gate.md` mode-3 and `docs/adr/0031-connect-wallet-gate-hides-when-disconnected.md`.
- `useIsWalletConnected` — returns `true` iff `ready && authenticated && walletReady`. Single source of truth for "User is usefully connected."
- `OnboardingFlowProvider` / `useOnboardingFlow` / `OnboardingFlowContext` / `OnboardingState` — drives the post-login onboarding FSM. `OnboardingState` (in `hooks/onboarding-flow.types.ts`) is the discriminated union `idle | resolving | onboarding | needs-handle{submitHandle} | needs-mfa{setupMfa,skipMfa} | needs-personalize{finishPersonalize} | ready{me} | error{message}`. After `getMe → onboard`, a new account routes to `needs-handle`; `submitHandle` (`POST /api/account/onboard` carrying the chosen handle) advances to `needs-mfa`; `setupMfa` (calls `enrollMfa`) or `skipMfa` reach `needs-personalize` (the new-account-only **Personalize** step — client-only theme + mobile trading-layout prefs, no server call); `finishPersonalize` then reaches `ready`. A returning user's real handle goes straight to `ready` (never entering `needs-personalize`). The `ready` variant also carries two optional canonical-`me` mutators (Workstream D — Selected-Wallet server↔FE desync fix): `applyMe(me: Me): void` replaces the cached `me` with a server-returned payload (no round-trip), and `refreshMe(): ResultAsync<Me, HttpError>` re-GETs `/api/account/me` and applies the result. They are produced by `useOwnOnboardingFlow` (which owns `me` state) and threaded through the provider, so both the own-flow and provider-supplied paths expose them; they are typed optional only so existing `{ kind: 'ready', me }` test literals stay valid. Neither retriggers the one-shot initial resolve (`startedRef`). The Account Modal's Wallets section feeds every select/import/remove's returned `Me` into `applyMe` (the canonical `me` is now the single source of truth), and `use-account-modal-content.ts` fires `refreshMe()` on the modal's closed→open transition so an out-of-band Selected-Wallet change (another device, a manual DB edit) reconciles. This keeps `useSelectedWallet` (reads `flow.me.wallets`) and the modal from diverging from the server `is_selected`.
- `getMasterViemAccount: (master: WalletAddress) => Promise<WalletClient | null>` on `AuthState` — exposes the **Selected Wallet** as a viem `WalletClient` (JSON-RPC account) so `hyperliquid/` can sign actions without importing `@privy-io/react-auth` directly (Option A signing boundary; ADR-0012, superseded to WalletClient; generalized by **ADR-0060**). Built inside `AuthBridge` (where `useWallets()` provides the `ConnectedWallet` list) and injected into `AuthValueProvider` as a prop so the latter stays testable without `PrivyProvider`. **Takes the target `master` address** — the resolved Selected-Wallet master from `useSelectedWallet().masterAddress`, which each consumer passes. The address is a parameter (not read inside `AuthBridge`) to avoid a circular dep: `useSelectedWallet` depends on `useAuth`. **Selection rule:** `findConnectedMasterWallet(wallets, master)` — the connected wallet whose address matches `master` case-insensitively, **regardless of wallet type** (ADR-0060 D-2 removed the `walletClientType !== 'privy'` filter). The embedded Native wallet is now a valid signing master — a selected embedded/imported wallet signs. Returns `null` when no connected wallet matches `master` (e.g. an injected wallet still re-hydrating). **Why WalletClient (not LocalAccount):** the `@nktkas` SDK calls `getChainId()` to derive `signatureChainId` for the EIP-712 domain; a `LocalAccount` falls back to chainId 1, causing `signTypedData` to fail on wallets active on other chains (e.g. Base = 8453). A `WalletClient` with the EIP-1193 provider exposes `getChainId()` so the SDK reads the real chain. See ADR-0012 "WalletClient supersession" section.
- `getBroadcastWalletClient: (master: WalletAddress) => Promise<WalletClient | null>` on `AuthState` (ADR-0028; generalized by ADR-0060) — the **transaction-broadcast** sibling of `getMasterViemAccount`. Returns a broadcast-capable viem `WalletClient` (can `writeContract` / `sendTransaction` / `wallet_switchEthereumChain`), bound to **Arbitrum One (chainId 42161)** via `chain: arbitrum` (`viem/chains`), for `hyperliquid/`'s deposit service to broadcast a plain ERC-20 `USDC.transfer` to the HL Bridge2 — a generic EVM write, **not** an `@nktkas/hyperliquid` SDK call (stays out of the four-file SDK lint zone). Built in `AuthBridge` alongside `getMasterViemAccount`, injected through `AuthValueProvider` as a prop, read via `useAuth()`. **Takes the same `master` address** (the resolved Selected-Wallet master). **Selection rule:** reused verbatim from `getMasterViemAccount` — `findConnectedMasterWallet(wallets, master)`, embedded included; returns `null` when none resolvable. The two accessors are siblings with distinct contracts (typed-data signing vs. broadcast) and must not be conflated. `account/` owns the wallet **capability** only — the chain-switch *policy* (when to switch, what to do on rejection) and all deposit logic live in the consuming venue's deposit service / state machine, never here (ADR-0028 D-3). The shared selection rule (used by both accessors **and** `isBroadcastWalletReady`) is the `findConnectedMasterWallet` util in `auth-provider.utils.ts` — one source for "the connected wallet matching the master address," not three inlined copies. The shared client factory is `buildMasterWalletClient` (the only difference between the two accessors is whether `chain: arbitrum` is bound).
- `getAgentWalletBroadcastClient: (agentWalletAddress: WalletAddress) => Promise<WalletClient | null>` on `AuthState` (**ADR-0082**) — a second, Agent-Wallet-scoped sibling of `getBroadcastWalletClient`, **not** a repurposing of it. Same builder (`buildMasterWalletClient`) and selection rule (`findConnectedMasterWallet`), but **takes the Agent Wallet's own address** — never the trading Selected-Wallet `master` — and is bound to **Base (chainId 8453)** via `chain: base`, where the Agent Wallet's USDC lives, instead of Arbitrum. Exists because the Agent Wallet is deliberately excluded from ever being a selectable trading master (see `exportableAddresses` above and `getMasterViemAccount`'s doc) — reusing `getBroadcastWalletClient` for an Agent Wallet withdrawal would sign with the wrong wallet entirely, which is exactly the bug this accessor fixes. Consumed by `agent-balance/`'s withdraw flow, paired with `switchMasterWalletChain(agentWalletAddress, BASE_CHAIN_ID)` + a post-switch re-verify (ADR-0080 D-3's idiom, reused verbatim) before broadcasting.
- `switchMasterWalletChain: (master: WalletAddress, chainId: number) => Promise<ChainSwitchOutcome>` on `AuthState` (**ADR-0080**) — switches the master wallet to `chainId` via Privy's native `ConnectedWallet.switchChain(chainId)`, resolved from `useWallets()` by the shared `findConnectedMasterWallet(wallets, master)` rule. This — **not** viem's `walletClient.switchChain` on the raw EIP-1193 provider — is the switch that actually takes effect for the embedded Native wallet: the raw-provider `wallet_switchEthereumChain` *resolves without switching* for embedded wallets, which is the "Switch to HyperEVM / Switch to Arbitrum" **dead-button** bug this replaces. Built in `AuthBridge`, injected through `AuthValueProvider`, read via `useAuth()`. Returns a neutral `ChainSwitchOutcome` (`'switched' | 'rejected' | 'failed'`) — a boundary-safe union so venue flows never import `AuthError`; `'rejected'` = the user declined (mapped from a `cancelled` AuthError), `'failed'` = no resolvable wallet or any other failure. `account/` owns the **capability** only — the switch *policy* (when to switch) and the mandatory post-switch chain-change **verification** (Privy does not update already-built providers, so a resolved switch is not proof of a chain change — ADR-0080 D-3) live in the consuming venue's deposit / EVM⇄Core state machine (ADR-0028 D-3 preserved). The `chainId` must be in Privy's `supportedChains` allow-list (`AuthProvider` registers `[arbitrum, hyperEvmChain, base]`) or Privy rejects it. Consumed by `hyperliquid/`'s deposit-flow ("Switch to Arbitrum"), evm-core-flow ("Switch to HyperEVM"), and `agent-balance/`'s withdraw flow ("Switch to Base", ADR-0082).
- `isBroadcastWalletReady: boolean` on `AuthState` — whether `findConnectedMasterWallet(wallets, …)` resolves **right now**, i.e. `getBroadcastWalletClient(master)` will return a client rather than null. A strictly stronger signal than `walletReady`: after a page load `walletReady` flips true on the canonical `user.wallet.address` while an injected wallet is still re-hydrating into `useWallets()`, so for a window `walletReady === true` but the broadcast wallet is unresolvable. Consumers that must broadcast a transaction (the HL deposit preflight) gate on **this**, not `walletReady`, to avoid a first-open `wallet-unavailable` abort. **ADR-0060:** keyed on the Privy-canonical `primaryAddress` with the `walletClientType !== 'privy'` filter removed — so for an embedded-only user the embedded wallet resolves here and the user is **no longer permanently gated** at the deposit preflight (was `false` permanently for email users before ADR-0060).
- `connectableMasterAddresses: ReadonlyArray<string>` on `AuthState` — the lower-cased addresses of **every** wallet currently live in the Privy session (those for which `findConnectedMasterWallet` would resolve a signer right now), **including the embedded one** (ADR-0060 D-2 removed the `!== 'privy'` filter). The **live-session signal** the Selected-Wallet → master reconciliation reads (Slice E / Slice 05 / ADR-0060). A selected embedded wallet now appears here, so `resolveSelectedMaster` returns the embedded address as the `masterAddress` and `isSelectionConnectable: true`.
- `useSelectedWallet(): SelectedWalletView` (`hooks/use-selected-wallet.ts`) — the **Selected-Wallet → master-address resolution + live-session reconciliation seam** (PRD-0006 Slice E). Composes the server-stored Selected Wallet (the `isSelected` wallet from `GET /me`, persisted across devices) with `connectableMasterAddresses` via the pure `resolveSelectedMaster` (`auth-provider.utils.ts`). Returns `{ selectedAddress, masterAddress, nativeAddress, isSelectionConnectable }` — all three addresses are branded `WalletAddress | null` (parsed via `parseWalletAddress`; consumers that re-key on it need the brand). `masterAddress` is the address the **grant/deposit/builder** master-signing path resolves to — the Selected Wallet **when it is currently connectable**, else **`null`** (ADR-0061 / Fix 3): a picked-but-not-connected imported wallet does **not** fall back to Native — signing consumers treat `null` as `signing-unavailable` (connect-to-grant), never a silent Native signature. `masterAddress` falls back to the Privy-canonical `primaryWalletAddress` **only when there is no stored selection at all**. `nativeAddress` is the **Native (embedded) wallet** (`selectNativeWallet(me)`, ADR-0061) — the account-stable identity the Hyperliquid agent key + name key on (one agent per account, independent of which wallet is selected). **Consumers pass `masterAddress` to the `getMasterViemAccount` / `getBroadcastWalletClient` accessors** (ADR-0060) — those accessors take the target master address rather than reading it themselves (avoids the circular dep, `useSelectedWallet → useAuth`). `isSelectionConnectable` is `false` when a server selection exists but is not in the live session (e.g. selected on another device / not yet re-connected): the UI **surfaces** this rather than silently signing with a non-live selection. **ADR-0060:** the embedded Native wallet is connectable, so a selected embedded master resolves as the `masterAddress` and an email-only Native-only User can sign + deposit + trade. **The Hyperliquid venue-readiness consumers read this hook:** `hyperliquid/`'s `useAgentWallet` / `useBuilderFee` / `useDeposit` read `masterAddress` for the grant + on-chain query, the agent **key/name** key on `nativeAddress` (`hl-agent-key:<nativeAddress>:<network>`, ADR-0061), and the **AI-suggestion execute path** + the Place-Order gate fire `(Selected Wallet × venue)` onboarding. Switching the Selected Wallet re-grants the **same** account agent without tearing down its key (no proliferation). Selection itself is driven by `api/select-wallet.ts`'s `POST /api/account/wallets/:address/select`.
- `useRecipientSuggestions({ selfAddress }): { walletSuggestions, recentSuggestions, recordRecipient }` (`hooks/use-recipient-suggestions.ts`) — recipient suggestions for a send/withdraw destination field: the user's own wallets (from the onboarding cache, `me.wallets`) via the shared `buildWalletRecipientSuggestions` builder, plus addresses the user recently sent to via the shared `createRecentRecipientsStore` (localStorage, keyed by Privy DID) and `buildRecentRecipientSuggestions`. Lives here because `account/` is the only module allowed to read the wallet list; `hyperliquid/`'s Send/Withdraw flows and `agent-balance/`'s Withdraw flow all consume this hook and feed the results into the shared `RecipientCombobox` (`shared/components/recipient-combobox`). `selfAddress` drops that address from the wallet-suggestion list (Send passes the Selected Wallet master to block self-send; Withdraw passes `null` since withdrawing to your own wallet is the default). `recordRecipient(address)` persists a completed destination so it surfaces under "Recent" next time.
- Domain types: `User`, `PrivyId`, `Wallet`, `Me`, `OnboardingInput`, `AuthError`, `ChainSwitchOutcome` (`'switched' | 'rejected' | 'failed'` — the neutral outcome of `switchMasterWalletChain`, ADR-0080), `WalletSource` (`'embedded' | 'external' | 'imported'` — ADR-0076 D-6 added `imported`: a raw private key the user imported into Privy's TEE-sharded infra, distinct from link-only `external`. The exportability discriminator: `embedded` + `imported` are owner-exportable, `external` is not). `OnboardingInput` carries an optional `inviteCode?: string` (sent only when the invite gate is on); `submitHandle(handle, inviteCode?)` threads it into `POST /api/account/onboard`.

`AuthError` is a discriminated union: `{kind:'no-credential'} | {kind:'cancelled'} | {kind:'not-configured'} | {kind:'unknown', cause: unknown}`. The connect-wallet modal hook holds the latest error and the modal renders a `role="alert"` line above the auth buttons; switching auth method clears it. Email-OTP failures from `useLoginWithEmail()` are surfaced through the same slot.

## Owns

- The single integration with `@privy-io/react-auth` for the whole client. No other module may import it directly.
- The `OnboardingFlowProvider` is mounted in `AppShell` (composition root), watches `ready && authenticated && wallet ready`, calls `getMe()`, and on 404 calls `onboard()`. It lives in `providers/onboarding-flow-provider/` (provider-unit folder); the consumer hook `useOnboardingFlow` stays in `hooks/use-onboarding-flow.ts` because it has both a context-aware path (used by ≥2 consumers under the provider) and a stand-alone path (`useOwnOnboardingFlow`) that the provider itself drives.
- The `OnboardingStepper` (email-only stepper, PRD-0006 UI-1) is mounted exactly once inside `AuthProvider`'s subtree (by `AuthValueProvider`). It is **not** exported from `index.ts` — its only driver is `openConnectModal()` / `isConnectModalOpen` on `useAuth()`. It renders inside the shared `Modal`, swapping the body per step (`Email → OTP → Handle → 2FA → Personalize`); steps 1-2 are the pre-auth Privy email-OTP flow (`useEmailOtpSteps`), and once authenticated the post-auth FSM (`useOnboardingFlow`) drives steps 3-5 for **new accounts only**. The existence fork is `GET /api/account/me`: a returning user (200) goes straight to `ready`; a new user (404) enters `needs-handle`. The final **Personalize** step (`PersonalizeStep` + `use-personalize-step`, `TOTAL_STEPS = 5`) is client-only: two `SegmentedControl`s for the app **theme** (`useThemeContext().toggleTheme`) and the mobile **trading layout** (`useTradingMode().setMode`) — both apply/persist live — plus a **Done** button (`finishPersonalize`). It requires `ThemeProvider` + `TradingModeProvider` as ancestors (present in production; the stepper renders under both via the router). **The handle is mandatory and collected before the account exists** (ADR-0058 amendment): the Handle step's submit IS the `POST /api/account/onboard` call carrying the chosen handle (`use-onboarding-flow.ts`'s `submitHandle`) — there is no provisional/default handle and no separate set-handle call. A 409 `HANDLE_TAKEN` from that onboard surfaces inline on the Handle step. Closing the modal at the Handle step creates **no** row. The modal closes when the FSM reaches `ready`. **Invite gate:** when enabled, the Handle step also shows an invite-code field (`use-handle-step.ts` reads `useAppConfig().inviteGateEnabled` from `shared/providers/app-config-provider`); the code is submitted with the handle, and an `INVALID_INVITE_CODE` (400) / `INVITE_CODE_ALREADY_REDEEMED` (409) — or a malformed-code `INVALID_REQUEST` — surfaces inline next to the field (via `inviteErrorFrom`), not as the handle-error toast. The gate flag is fetched once post-auth from the public `GET /api/config` (`shared/api/get-app-config.ts`), defaulting fail-safe to enabled. `inviteErrorFrom` / `submitErrorToast` (`onboarding-stepper.utils.ts`) read the server `error.code` / `issues` via the shared `apiErrorCode` / `apiErrorIssues` extractor (`shared/utils/api-error-copy`, PRD-0007) — the `{ error: { code, … } }` envelope is no longer parsed locally; the user-facing copy stays module-local.

## Provider-unit layout

Both providers follow the provider-unit pattern (`apps/client/.claude/rules/frontend-architecture.md` → "Provider unit pattern"):

- `providers/auth-provider/` — `AuthProvider.tsx` + `auth-provider.context.ts` + `use-auth.ts` + `__tests__/` + `__fixtures__/auth-bridge-for-test.tsx` + `index.ts`. The fixture re-exports the inner `AuthValueProvider` as `AuthBridgeForTest` so tests can drive the context value without mounting `<PrivyProvider>`; production code never imports the fixture.
- `providers/onboarding-flow-provider/` — `OnboardingFlowProvider.tsx` + `onboarding-flow-provider.context.ts` + `index.ts`. The consumer hook `useOnboardingFlow` is in `hooks/use-onboarding-flow.ts` (see "Owns" above).
- `providers/account-modal-provider/` — `AccountModalProvider.tsx` + `account-modal-provider.context.ts` + `use-account-modal.ts` + types + `index.ts`. Holds the Account Modal's `{ isOpen, open, close }` controller (Slice 04). Mounted in `AppShell` so the header `AccountAvatarTrigger` (open) and the `AccountModal` (read `isOpen` / `close`) share it. **Auto-closes** when `useAuth().authenticated` goes `false` (logout or session loss) — the modal is auth-gated UI and must not linger over the disconnected shell. `AccountModalContext` is exported through the barrel for test wrappers that mount it with fixture state.

Contexts are exported through the module barrel (`AuthContext`, `OnboardingFlowContext`) because cross-component test wrappers and `app/app-shell/__tests__/routing.test.tsx` mount them by hand with fixture state. New production code reads via `useAuth` / `useOnboardingFlow`, not the contexts directly.

## Depends on

- `@privy-io/react-auth` — the wrapped identity provider (only here). `AuthBridge` imports `useMfaEnrollment` and reads `user.mfaMethods` to derive `hasMfa` (the binary 2FA signal — whether `'totp'` is in `user.mfaMethods`). For ADR-0076 it also imports `useExportWallet` (→ `exportWallet({ address })`, behind `AuthState.exportWallet`) and `useImportWallet` (→ `importWallet({ privateKey })`, behind `AuthState.importPrivateKey`). For ADR-0078 it also imports `useCreateWallet` (→ `createWallet({ createAdditional: true })`, behind `AuthState.createAgentWallet`) and `useSigners` (→ `addSigners` / `removeSigners`, behind `AuthState.attachAgentSigner` / `removeAgentSigner`) — the Agent Wallet create + scoped-signer seams `agent-balance/` consumes. The `exportableAddresses` array is derived in `AuthBridge` from `useWallets()` (the `walletClientType === 'privy'` subset, which now includes the user-owned Agent Wallet). These are the **only** Privy export/import/create/signer callsites in the app.
- `hyperliquid` (public barrel: `@/modules/hyperliquid`) — `AuthProvider.tsx` imports `resolveHyperEvmChain` and adds the resolved HyperEVM `Chain` to Privy's `supportedChains` array (previously `[arbitrum]`, now `[arbitrum, hyperEvmChain]`). Without it, Privy rejects the `switchChain`/`addChain` calls the EVM⇄Core flow issues, causing the "Switch to HyperEVM" step to silently no-op (ADR-0069). The chain is resolved inside the component via `useMemo` (not at module scope) to avoid a circular init crash.
- `web3-avatar-react` — the default account avatar (deterministic gradient seeded from the Native Wallet address). Imported only by `components/account-avatar/Web3Avatar.tsx`; not a stack swap (PRD-0006 CS-8). The `<Avatar>` self-injects its CSS and defaults to a circular skin — `Web3Avatar` overrides it square (`border-radius: 0`) to match the pixel-art system.
- `viem` / `viem/accounts` — `getAddress()` for EIP-55 checksumming of wallet addresses; `WalletClient`, `createWalletClient`, `custom` for the `getMasterViemAccount` implementation and return type.
- `shared/http` — the authenticated HTTP transport. `account/` instantiates it inside `AuthProvider` (passing `getAccessToken`) and exposes the resulting `ApiClient` on `useAuth().apiClient`. Endpoint wrappers in `account/api/` (`getMe`, `onboard` — now carries the chosen `handle`, `selectWallet`, `checkHandleAvailable`, and the Slice-06 `importWallet` / `removeWallet`) are thin and call `client.get` / `client.post` / `client.delete`. ## Cross-app contract

- `POST /api/account/onboard` (creates `users` row + `wallets` rows) and `GET /api/account/me` against server `account` module. Both carry `Authorization: Bearer <Privy JWT>`. See `apps/server/src/account/MODULE.md` and `/CLAUDE.md` "Auth contract".
- Token-refresh + retry semantics: a `TOKEN_EXPIRED` / `TOKEN_INVALID` response triggers exactly one refresh-and-retry; second failure surfaces `SessionExpiredError`. Implemented in `shared/http`; `AuthProvider` subscribes to expiry and triggers a **stale-session reset**: `await logout()` then `clearPrivyStorage()` (`providers/auth-provider/clear-privy-storage.ts`) purges every `privy:`-prefixed `localStorage` key. This self-heals a dead/`TOKEN_INVALID` token that Privy's own `logout()` can't clear — e.g. a key left by a previously configured `VITE_PRIVY_APP_ID` — so the next login starts clean without a manual "Clear site data". **Also**, `use-auth-value` observes Privy's `authenticated` flipping `true → false` directly: Privy can drop a session on its own (stale/unrecoverable refresh token — e.g. the Privy user was deleted) with **no 401 reaching `shared/http`**, so the subscription never fires. The observer surfaces the same `session-expired` toast on that transition, **skipping it when the user logged out deliberately** (the exposed `logout` sets a `deliberateLogoutRef` first). The toast `id` de-dupes against the `shared/http` path when both fire for one loss.

## Gotchas

- **No `AccountAdapter` port — by design.** Account is identity, not a Venue. There is no mock implementation, no port/adapter pair. Tests stub `fetch` (or use MSW). Re-introducing the port "for symmetry with `trading/`" is cargo-culting. See `docs/adr/0004-no-account-adapter-port.md`.
- **`AppShell` mounts `OnboardingFlowProvider`, not pages.** Don't add onboarding logic inside `TradingPage` / `PortfolioPage` — by the time a page mounts, onboarding has already resolved.
- The `User` identifier across the entire system is the **Privy DID** (`did:privy:...`). Don't mint surrogate ids.
- **Primary Wallet = `user.wallet.address`, never `useWallets()[0]`.** The opaque onboard 403 (and the follow-on "stuck loading") was caused by onboarding with `wallets[0]` — for a MetaMask user that is the currently-selected injected account, a different address than the one Privy linked. The server's `verifyWalletOwnership` checks Privy's `linkedAccounts`, which contains `user.wallet.address`, not the arbitrary `wallets[0]`. `AuthBridge` now derives `primaryAddress`/`primaryClientType` from `user.wallet`. Keep display, onboarding, and signing (`getMasterViemAccount`) all keyed off `user.wallet.address`.
- **Onboarding gates on `canStart` = connected + `privyId` + `walletAddress`.** The run-effect cleanup resets `startedRef` so a run cancelled by a transient `canStart` drop can restart (otherwise the flow froze at `resolving`). The server logs which `Forbidden` branch fired (`apps/server/src/account/account.service.ts`).
- **DEV-only mock-auth seam (`VITE_MOCK_AUTH`).** `account.config.ts` → `resolveMockAuthConfig(import.meta.env)`; when it returns non-null (`import.meta.env.DEV && VITE_MOCK_AUTH === 'true'`), `AuthProvider` renders `MockAuthBridge` instead of `PrivyProvider`, reporting a connected mock wallet so headless verification (Playwright / Spectate Mode) can render. Dead-code-eliminated in production builds. NOT a port (ADR-0004 holds). See `docs/adr/0055-dev-only-mock-auth-toggle.md`.

## Out of scope

- Trading or portfolio data — those modules have their own surfaces.
- Wallet-balance reads, transaction submission. (Privy handles the wallet; this module exposes the User and Onboarding, plus the Option A signing bridge `getMasterViemAccount`.)
- The HTTP-layer schema validation — the server is the source of truth for request bodies (`onboard.dto.ts` server-side).
