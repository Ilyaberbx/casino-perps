# hyperliquid (client)

> **Scope of this file:** non-obvious context only. Update in the same PR that changes the public surface, what the module owns, or what it depends on.

## Purpose

The first non-mock **Venue** — implements the capability set defined in ADR-0008 against the Hyperliquid exchange. Capabilities shipped: `connection`, `marketData`, `candles`, `portfolio`, `balances`, `accountMode`, `equityExtensions`, `marginSummary`, `feeSchedule`, `volumeHistory`, `perpsPositionsSnapshot`, `openOrdersSnapshot`, `twapActiveSnapshot`, `twapHistory`, `twapController`, `twapSliceFills`, `tradeHistory`, `orderHistory`, `interestHistory`, `accountActivity`, `trader` (with `supportsTriggerOrders` / `supportsStopOrders` / `supportsTwap` all `true` — Market/Limit plus the Stop Market / Stop Limit / TWAP Pro types, ADR-0034 D-3), `leverageController`, `marginModeController`, `positionProtection`, and the Acting-Address-keyed `ownAccount` group (`portfolio` / `balances` / `perpsPositionsSnapshot` / `feeSchedule` / `accountMode`; ADR-0038 — `accountMode` added so the Manage Funds Transfer flow's Perps⇄Spot applicability check never reads the Spectated Address's mode). The reconnect/state-machine design from ADR-0009 has been superseded by ADR-0010 (the SDK owns reconnect; we surface SDK signals).

## Public surface

- `createHyperliquidVenue(options: HyperliquidVenueOptions): Venue` — factory. Threads `getAddress: () => WalletAddress | null` (the **Viewing Address** — `spectate ?? connected`) so wallet rotation does not rebuild the venue, an optional `getActingAddress: () => WalletAddress | null` (the **Acting Address** — connected-only; defaults to `getAddress` so mock-venue/tests with no spectate concept see acting === viewing and the order data is unchanged), an optional `getAgentWallet: () => HyperliquidAgentWallet | null` (the agent signing-wallet bridge — see "Agent signing-wallet contract"; omitted ⇒ `trader` still declares but every signed action returns a typed `rejected` error), plus a required `logger: Logger` (bound module: `hyperliquid-venue`) which the factory fans out to gateway, stream, pull, and the readers. **Order-flow + Manage Funds keying (ADR-0038):** the venue exposes `capabilities.ownAccount: { portfolio, balances, perpsPositionsSnapshot, feeSchedule, accountMode }` keyed to the Acting Address. The trader's `validateDraft`/`previewOrder` holders + `getCurrentLeverageState` read the first four — so order entry, validation/preview, the max slider, the fee estimate, and the leverage seed reflect the User's own account while Spectating. The Manage Funds flow providers (`providers/{withdraw,transfer,send,deposit,evm-core}-flow-provider/`) read `ownAccount.balances`/`ownAccount.portfolio` for their withdrawable/sendable/transferable caps + deposit-credit detection, and `TransferFlowProvider` reads `ownAccount.accountMode` for the Perps⇄Spot applicability check — none of it ever reflects the Spectated Address. The acting account sources lazily diverge from the viewing sources only while spectating (when acting === viewing they alias the viewing instances → zero extra subscriptions; D-2). `refreshAddress()` re-keys the **Viewing** sources AND re-routes the acting selectors (spectate enter/leave may flip divergence); `refreshActingAddress()` re-keys the **Acting** sources (wallet rotation) — the composition root calls BOTH from its address-mirror effect, `refreshAddress()` only from `SpectateBridge` (D-3). **Construction is pure** — it performs no I/O and starts no streams/timers (it no longer calls `marketData.refresh()`). The returned `Venue` exposes `dispose(): void` (idempotent; composes `stream.stop()` + `pull.stop()` + the acting sources' `stop()` + the trader's `orderRefResolver.stop()`). The composition root owns the lifecycle: a `useEffect` calls `marketData.refresh()` on mount and `venue.dispose()` on cleanup. See ADR-0015. `venue.metadata.explorerTxUrl(hash)` returns the public-explorer URL — `https://app.hyperliquid.xyz/explorer/tx/<hash>` on mainnet, `https://app.hyperliquid-testnet.xyz/explorer/tx/<hash>` on testnet — sourced from `HYPERLIQUID_EXPLORER_BASE_URL`. `services/market-data-reader.ts`'s `projectTrade` surfaces `Trade.transactionHash = t.hash` (the WS `TradesEvent`'s per-trade `hash: 0x<...>` is identical to `RecentTradesResponse[i].hash`) and the participant pair `Trade.makerAddress = t.users[0]` / `Trade.takerAddress = t.users[1]` (the SDK trade event's `users` tuple is positional `[maker, taker]`; the taker is the aggressor whose `side` labels the trade — the mapping is by tuple position, not re-derived from `side`; both branded via `brandSdkAddress`, left `undefined` when `users` is absent); the trades-tape consumes all three to render the per-row TX link and the Taker/Maker columns.
- `loadHyperliquidConfig(env): Result<HyperliquidConfig, HyperliquidConfigError>` — env parser; called by `app/` at startup. On `err`, the venue selector hides Hyperliquid (no silent fallback).
- `HYPERLIQUID_VENUE_ID`, `HYPERLIQUID_VENUE_LABEL` — stable identifiers.
- `resolveHyperEvmChain(env): Chain` — resolves the viem `Chain` for HyperEVM (chain id 999 mainnet / 998 testnet) from the build env. Re-exported from `services/hyperevm.config.ts` for `account/`'s `AuthProvider` to add HyperEVM to Privy's `supportedChains` allow-list; without it, Privy rejects the `switchChain`/`addChain` calls the EVM⇄Core flow relies on (ADR-0069).
- Types: `HyperliquidConfig`, `HyperliquidNetwork`, `HyperliquidConfigError`, `HyperliquidVenueOptions`.
- `AgentWalletProvider` — React provider that owns the agent wallet status, keypair lifecycle, and `approveAgent` flow. Self-contained: constructs its own exchange gateway internally. Must be mounted inside `OnboardingFlowProvider`. No props. **Account-scoped, native-keyed agent (ADR-0061):** the agent **key + name** key on the account's **Native (embedded) wallet** (`useSelectedWallet().nativeAddress`) — `hl-agent-key:<nativeAddress>:<network>` — so there is **one agent per account**, stable across selected-wallet switches. The on-chain `queryAgents` and the `approveAgent` **grant** key on the connectable **Selected Wallet master** (`useSelectedWallet().masterAddress`): the selected wallet *grants* the account's agent access. `approve()` **always mints a fresh keypair** (ADR-0077 — Hyperliquid enforces anti-replay on agent addresses: reusing the stored key re-submits an already-used address and is rejected with *Extra agent already used*; this reverts ADR-0061 D-1). Proliferation is bounded by the **stable account-keyed name** (ADR-0061 D-2 / ADR-0036 D-1 — a re-grant on the same master *replaces* the named slot rather than minting a new one), not by key reuse. A stored key that still *matches* a live on-chain agent resolves to `'approved'` at bootstrap and skips re-approval, so an already-onboarded single-master user sees no churn. `masterAddress` is `null` when the selection is not connectable (Fix 3) → `approve()` returns `signing-unavailable` (connect-to-grant), never signs as Native. Switching the Selected Wallet mid-session re-evaluates readiness against the new master's grant **without** regenerating the key (the re-bootstrap is driven by `use-session-bootstrap`'s `bootstrapKey = masterAddress`). Bootstrap (#167, reworked by ADR-0036 D-2) reads the keystore, calls `gateway.queryAgents(user)`, stores the result in `existingAgents` state, and resolves via `resolveAgentDesync(storedKey, hlAgents, expectedAgentName, maxNamedAgents)`: local key matches an HL agent → `'approved'`; an HL agent carries OUR default name (`deriveDefaultAgentName`, now in `hyperliquid.utils.ts`) but the key is absent/mismatched → `{ kind: 'error', reason: 'agent-exists-no-local-key' }` (one-click same-name replace — HL replaces an agent when a named wallet with a matching name is re-registered); all `HYPERLIQUID_MAX_NAMED_AGENTS` (3) named slots foreign-owned and key matches none → `{ kind: 'error', reason: 'agent-slots-full' }` (victim picker); anything else (free slot, foreign agents only) → `'missing'` — a foreign agent with a free slot is NOT an error. A `queryAgents` network failure collapses to `'unknown'` rather than blindly trusting the local key (which would re-introduce the desync the pre-flight is designed to detect). A reactive `approveAgent` cap rejection maps to `agent-slots-full` and fire-and-forget re-queries `queryAgents` to refresh `existingAgents`.
- `useAgentWallet()` — consumer hook returning `{ status: AgentWalletStatus; agentAddress: WalletAddress | null; existingAgents: ReadonlyArray<HyperliquidKnownAgent> | null; approve: (agentName: string) => ResultAsync<void, AgentApprovalError>; getSigningWallet: () => HyperliquidAgentWallet | null }`. `AgentWalletStatus` starts at `'checking'` (initial/post-disconnect) until the bootstrap keystore+`queryAgents` desync resolution settles — distinct from `'missing'` (= "checked, no usable agent"); `'checking'` projects to the `'running'` step status and keeps the venue `bootstrapping` (see `HyperliquidOnboardingProvider`). `existingAgents` (ADR-0036) is the public `extraAgents` projection — `{ address, name, validUntil }`, never keys — that feeds the slots-full victim picker; `null` until bootstrap resolves, reset on disconnect. Throws if used outside `AgentWalletProvider`. `getSigningWallet` is the **stable** signing-wallet getter the app-scope signing-wallet holder registers (see "Agent signing-wallet contract"): it reads the in-memory private-key ref + a live `approved` flag and builds the viem `AbstractWallet` lazily, per call, at signing time — the key never enters React state, the holder, the venue, logs, or serialization. Returns `null` unless an agent is `approved`. `AgentApprovalErrorReason` widened in #166, #167, and Phase 07 Plan 01; **reworked by ADR-0036**: `'agent-cap-reached'` was REPLACED by `'agent-slots-full'` (both the proactive bootstrap detection and the reactive gateway `agent-cap-reached` rejection converge on it — `gatewayKindToAgentReason` does the mapping). Current union: `'wallet-rejected' | 'chain-mismatch' | 'signing-unavailable' | 'corrupted-key' | 'agent-slots-full' | 'name-collision' | 'rate-limited' | 'deposit-required' | 'agent-exists-no-local-key' | 'keystore-write-after-approval' | 'agent-address-reused' | 'unknown' | 'approval-failed'` (`agent-address-reused` is the ADR-0077 HL anti-replay classification — gateway kind `agent-address-reused` from `/extra agent already used/i`, mapped to a self-healing labelled-retry CTA instead of `'unknown'`; effectively unreachable now that `approve()` always mints a fresh key. `approval-failed` is a legacy alias collapsing to the same CTA as `'unknown'`). Each reason maps 1:1 to a `{ headline, copy, cta }` tuple in `providers/hyperliquid-onboarding-provider/error-to-cta-mapping.ts`; `keystore-write-after-approval` uses the `reload-page` CTA (copy corrected by ADR-0036 D-6 — a same-name re-approve replaces, it cannot duplicate). **No agent or builder reason maps to an external-link CTA** (ADR-0036 D-5 — `agent-exists-no-local-key` is a labelled-retry "Replace agent" one-click same-name replace; `agent-slots-full` is a labelled-retry "Replace selected agent" backed by the victim picker; structurally asserted in the mapping test).
- `BuilderFeeProvider` — React provider that owns the partner-builder fee approval status against `HYPERLIQUID_BUILDER_ADDRESS` (ADR-0024). Self-contained: constructs its own exchange gateway internally. Must be mounted inside `AgentWalletProvider` (and therefore inside `OnboardingFlowProvider`). No props. **Selected-Wallet keying (Slice 07):** like the agent provider, the `queryMaxBuilderFee` bootstrap + `approve`/`replaceBuilder` key on `useSelectedWallet().masterAddress`, re-querying for the new account on a selection switch (via `use-session-bootstrap`'s `bootstrapKey`), never the single Primary Wallet. Bootstraps from `gateway.queryMaxBuilderFee(user)` on session start; status `'approved'` when the on-chain rate is `≥ HYPERLIQUID_BUILDER_FEE_TENTHS_OF_BPS`, `'missing'` otherwise. `builderFee.approve()` is invoked **only** through the onboarding hook's `runAll`/`retryStep` (both guarded by `_isRunning`) — there is no longer a concurrent direct caller from `account/`. `approveBuilderFee` is NOT idempotent (it submits a signed on-chain action). The guard against a double-sign is that `approve()` has a **single serialized caller** — onboarding `runAll`/`retryStep`, mutually excluded by `_isRunning` (set synchronously, no await gap). The render-captured completeness pre-check in `runAll` does NOT defend against a *concurrent external* caller: it reads a `useCallback`-closed `builderStepStatus`, so a status flip by another consumer mid-run would not be seen. CR-01 is closed only because no such second consumer exists today. **If the deferred order-placement contract (below) or a revived `account/` bundled-signing path adds a second consumer of `approve()`, re-open CR-01 and gate each leg on a live-status ref instead of the captured projection.**
- `useBuilderFee()` — consumer hook returning `{ status: BuilderFeeStatus; approvedBuilders: ReadonlyArray<WalletAddress> | null; approve: () => ResultAsync<void, BuilderFeeApprovalError>; replaceBuilder: (victimBuilder: WalletAddress) => ResultAsync<void, BuilderFeeApprovalError> }`. Throws if used outside `BuilderFeeProvider`. `BuilderFeeStatus = 'checking' | 'approved' | 'missing' | 'approving' | { kind: 'error', reason: BuilderFeeApprovalErrorReason }` — `'checking'` is the initial/post-disconnect state until the bootstrap `queryMaxBuilderFee` read settles (distinct from `'missing'` = "checked, fee not approved"); it projects to the `'running'` step status and keeps the venue `bootstrapping` (see `HyperliquidOnboardingProvider`). `BuilderFeeApprovalErrorReason` widened in #166 to `'wallet-rejected' | 'chain-mismatch' | 'signing-unavailable' | 'builder-not-funded' | 'approval-cap-reached' | 'rate-limited' | 'unknown'`, and widened in Phase 07 Plan 01 with `'deposit-required'` (gateway signals account unfunded before the `approveBuilderFee` action — same error path as agent approval) (plus `'approval-failed'` as a legacy alias collapsing to the same CTA as `'unknown'` until sibling worktree #172 lands). **ADR-0036 D-4 (builder-cap recovery, reactive):** when an approve rejects with `approval-cap-reached` (HL caps a user at 10 active builder approvals), the provider fire-and-forget fetches `gateway.queryApprovedBuilders(user)` into `approvedBuilders` state (our own `HYPERLIQUID_BUILDER_ADDRESS` filtered out) — there is NO bootstrap-time builders query. `replaceBuilder(victim)` chains two master-wallet signatures: `revokeBuilderFee(victim)` (a `maxFeeRate: "0%"` re-approval — HL's only slot-freeing mechanism) then the normal `approveBuilderFee`. If the second leg fails the slot is already freed, so a plain retry completes. The cap-rejected step's CTA is a labelled retry ("Revoke & approve") backed by a full-address picker — no external link. The bootstrap path (`queryMaxBuilderFee` info read) shares the same SDK-error→reason taxonomy as the approve path — no separate `'bootstrap-failed'` reason. The `approve()` call requires a master `WalletClient` resolved via the ADR-0012 `getMasterViemAccount` callback — **passing the Selected-Wallet master (`useSelectedWallet().masterAddress`, embedded included per ADR-0060)** — and signs an `approveBuilderFee` user-signed action with `maxFeeRate = HYPERLIQUID_BUILDER[network].maxFeeRate` and `builder = HYPERLIQUID_BUILDER_ADDRESS`.
- `DepositProvider` — React provider that owns the **First Deposit** milestone state for the current wallet (Phase 07 Plan 03 / DEP-05; reframed by ADR-0027 + amendment). Self-contained: constructs its own exchange gateway internally. No props. **Selected-Wallet keying (PRD-0006 Slice 07):** First Deposit is step 0 of venue onboarding, so its `queryHasEverFunded` bootstrap + `recheck()` key on `useSelectedWallet().masterAddress` (the same `(Selected Wallet × venue)` gate as agent/builder), re-querying for the new account on a selection switch (via `use-session-bootstrap`'s `bootstrapKey`), not the single Primary Wallet. Bootstraps from `gateway.queryHasEverFunded(user)` on session start — read-only, no signature required (D-7, Pitfall 4). **First Deposit is chain-derived ("ever funded" = non-funding ledger non-empty), not a live balance**: `'funded'` means the Account has ever been funded by any means (deposit/transfer/vault return) and does NOT reopen on withdrawal (the live `accountValue > 0` check is **Tradeable Funds**, a separate order-submit gate — see `shared/components/TradeableFundsGateButton`). Status: `'checking'` (bootstrapping) → `'funded'` (ok(true) — ledger non-empty) | `'needs-deposit'` (ok(false) — never funded / empty ledger) | `{ kind: 'error', reason: DepositErrorReason }` (genuine query failure). Exposes `recheck()` affordance — manual "I've deposited — re-check" (one-shot re-run of queryHasEverFunded, no WebSocket, no signature; covers the deposit settlement lag). Intended to be consumed by the onboarding step machine (Plan 04) as step-0.
- `useDeposit()` — consumer hook returning `{ status: DepositStatus; recheck: () => void }`. Throws if used outside `DepositProvider`. `DepositStatus = 'checking' | 'funded' | 'needs-deposit' | { kind: 'error', reason: DepositErrorReason }`. `DepositErrorReason = 'deposit-required' | 'rate-limited' | 'unknown'` (intentionally narrow — the deposit query is read-only; wallet-rejected/chain-mismatch/etc. collapse to `'unknown'` per LOW-1).
- `HyperliquidOnboardingProvider` — React provider that composes `DepositProvider` + `AgentWalletProvider` + `BuilderFeeProvider` into a single venue-agnostic `VenueOnboarding` value (ADR-0026 / Phase 07 Plan 04 / DEP-05). Must be mounted **inside** `<DepositProvider>`, `<AgentWalletProvider>`, and `<BuilderFeeProvider>` (consumes all three hooks). Pure aggregator — does not own funded-state/signing/keystore logic. No props. **Mount order in `AccountSessionRoot`:** `OnboardingFlowProvider > DepositProvider > AgentWalletProvider > BuilderFeeProvider > VenueOnboardingSession`. The active Hyperliquid venue exposes this pair as its `Venue.onboarding` capability via `app/venues.ts`. **3-step onboarding (deposit→agent→builder):** step-0 is the deposit step (capability-less, D-9 — no `capability` key, no `inputs`). **HIGH-1 error-status CTA mechanism:** `needs-deposit` (and genuine deposit query errors) are projected to a `VenueOnboardingStepErrorStatus` (`kind:'error'`) carrying the **`open-deposit` CTA** (in-app — opens the shared `deposit-sheet-provider` via the `openDeposit` host action wired in `app/app-shell/AppShell.tsx`; was an external-link to `app.hyperliquid.xyz` before `.design/hyperliquid-deposit/` Chunk 4), because `StepRow.tsx` renders a step's CTA ONLY inside its error sub-card — a plain `pending` step would show NO deposit affordance at all. **ADR-0027 milestone unchanged:** `queryHasEverFunded` still drives step completion; only the affordance moved in-app. This is a per-STEP projection; it does NOT affect the venue-level status. **D-10 (never-blocked):** venue status never emits `{ kind:'blocked' }` — that would suppress the `bootstrapping→incomplete` auto-open the never-funded user depends on. Venue state machine: `bootstrapping` → `incomplete` (≥1 of 3 steps not complete) → `ready` (all three complete). **Reload semantics:** `bootstrapping` spans not just the pre-session-live window but the entire initial-check window — the venue stays `bootstrapping` until **all three** step providers leave their `'checking'` state (deposit/agent/builder bootstrap reads resolve asynchronously and independently). An already-onboarded reload therefore transitions `bootstrapping → ready` **directly**, with no transient `incomplete` — which is what previously flickered the progress badge 0/3→3/3 and fired a spurious "setup complete" toast. `incomplete` is reached only once all checks have settled with ≥1 step still outstanding, and the completion toast (`app/app-shell/use-onboarding-completion-effect.ts`) fires only on the `incomplete → ready` edge (genuine in-session onboarding), never `bootstrapping → ready`. `runAll(values)` gates on deposit complete before agent/builder signing (T-07-11). `retryStep('deposit', values)` calls `deposit.recheck()` — no signing (D-7/Pitfall 4). `retryStep(agent/builder, values)` re-fires the respective provider's approve. All operations guarded by `_isRunning`.
- `useHyperliquidVenueOnboarding()` — consumer hook returning the `VenueOnboarding`-shaped value. Throws if used outside `HyperliquidOnboardingProvider`. Wired into the generic `<VenueOnboardingProvider>` by `app/`.
- `TransferFlowProvider`, `useHyperliquidTransfer`, `TransferFlow` — the HL implementation of the venue-agnostic `VenueTransferCapability` port (ADR-0033 / slice 04). `TransferFlowProvider` owns the **Spot↔Perp transfer state machine** (`idle → signing → success | error`; typed errors `amount-invalid | insufficient-balance | wallet-rejected | deposit-required | rate-limited | network | unknown`). It is self-contained: builds its **own exchange (signing) gateway** for the master-wallet-signed `usdClassTransfer` action (mirrors `BuilderFeeProvider`), reads `getMasterViemAccount` from `useAuth()` **+ the resolved Selected-Wallet master from `useSelectedWallet().masterAddress` (ADR-0060 — signs as the Selected Wallet, embedded included; passes the master to the accessor)** (ADR-0012 / ADR-0033 D-2 — a user-signed action, NOT the agent path), the opening `prefill` + `close` from `useTransferSheet()`, and consumes the **active venue's live `balances` reader** (USDC `available` per account — Spot→Perp MAX = spot USDC row `available`; Perp→Spot MAX = perps USDC row `available`/withdrawable) and `accountMode` reader (drives `isApplicable = isSegregated`). `submit` validates client-side (amount > 0, ≤ available, ≤ 6 decimals) before signing; on `status:ok` it is **optimistic** — a success toast (`shared/services/toast`) + sheet close, no settlement polling (the live `webData2` stream reflects the new balances). On error it renders an inline `Callout` + retry with the amount + direction preserved. `useHyperliquidTransfer()` projects the rich state to the thin port `TransferState { isApplicable, isComplete: phase === 'success' }`. **The HL venue's `transfer` slot is composed in `app/venues.ts`** (mirrors `deposit`) — `createHyperliquidVenue` stays free of provider/React imports. mock-venue omits `transfer`, so no transfer affordance renders for it.
- `WithdrawFlowProvider`, `useHyperliquidWithdraw`, `WithdrawFlow` — the HL implementation of the venue-agnostic `VenueWithdrawCapability` port (Withdraw-to-Arbitrum; mirrors the transfer slice). `WithdrawFlowProvider` owns the **withdraw state machine** (`form → review → signing → sent | error`; typed errors `amount-invalid | insufficient-balance | destination-invalid | wallet-rejected | deposit-required | rate-limited | network | unknown`). It is self-contained: builds its **own exchange (signing) gateway** for the master-wallet-signed `withdraw3` action (mirrors `TransferFlowProvider`), reads `getMasterViemAccount` from `useAuth()` + the resolved Selected-Wallet master from `useSelectedWallet().masterAddress` (ADR-0012 / ADR-0060 — a user-signed action, NOT the agent path; the master also prefills the destination + drives `isApplicable = masterAddress !== null`), consumes the **active venue's live `balances` reader** + `accountMode` to derive the withdrawable cap **mode-aware** (`selectWithdrawableUsdc`, hyperliquid-account-modes.md §1/§3): **segregated** → `'perps'`-scope USDC `available` (the perp `withdrawable`); **unified / portfolio margin** → `'all'`-scope USDC `available` (the single unified collateral pool). The `'perps'` scope is deliberately EMPTY for unified accounts, so reading only it stranded unified users at `$0` (Withdraw is their only exit — Transfer is hidden for unified); it now subscribes to BOTH scopes + `accountMode` like the Transfer flow. Reads `account/`'s `useRecipientSuggestions({ selfAddress: null })` for the destination field's your-wallets + recent suggestions (every wallet offered, since withdrawing to your own wallet is the default). The two-step form gates `review()` on amount valid (`> 0`, `≥ MIN_WITHDRAW_USDC` (2), `≤ withdrawable`, `≤ 6` decimals) + destination valid (`parseWalletAddress`) + — only when the destination is **edited away from the user's own wallet** — an irreversible-confirm checkbox. The destination field is the shared `RecipientCombobox` (`shared/components/recipient-combobox`), wired by the colocated `use-withdraw-flow-body.ts` (which also owns the combobox's local open/close UI state via `useRecipientCombobox`). `submit` signs `withdraw3({ destination, amount })`; on `status:ok` it is **optimistic** (a success toast + the `sent` arrival track showing the flat `WITHDRAW_FEE_USDC` ($1) fee, `netReceived = amount − fee`, and the ~5 min Arbitrum estimate; the user dismisses with "Done" → `reset`) and records the destination via `useRecipientSuggestions`'s `recordRecipient` so it surfaces under "Recent" next time. On error it renders an inline `Callout` + retry with the amount + destination preserved. `useHyperliquidWithdraw()` projects to the thin port `WithdrawState { isApplicable, isComplete: phase === 'sent' }`. **The HL venue's `withdraw` slot is composed in `app/venues.ts`** (mirrors `transfer`) — `createHyperliquidVenue` stays free of provider/React imports. mock-venue omits `withdraw`, so no withdraw affordance renders for it.
- `SendFlowProvider`, `useHyperliquidSend`, `SendFlow` — the HL implementation of the venue-agnostic `VenueSendCapability` port (Send USDC/spot to an external Hyperliquid address; mirrors the withdraw slice). `SendFlowProvider` owns the **send state machine** (`form → review → signing → sent | error`) and self-wires its deps: builds its own master-wallet exchange gateway for the user-signed `usdSend` / `spotSend` actions, reads `getMasterViemAccount` + `useSelectedWallet().masterAddress` (ADR-0012; the master drives the **self-send guard** and `isApplicable`), composes a **sendable-token list** from the active venue's live `balances` reader + `accountMode` capability. USDC's route is **account-mode-aware** (hyperliquid-account-modes.md §0/§3): a **segregated** account sends perp USDC via `usdSend` (`'perps'`-scope `available`); a **unified** account sends its spot-held pooled USDC via `spotSend` (the `'all'`-scope `source:'unified'` USDC row) — reading only the perps scope stranded unified users at $0. Held spot rows send via `spotSend`. The list is joined to the spot-token id map (`"NAME:0xTOKENID"`, now including USDC's id) read once from `getSpotMetaAndAssetCtxs`, and reads `account/`'s `useRecipientSuggestions({ selfAddress: masterAddress })` for the destination field's your-wallets + recent suggestions — the Selected Wallet master is dropped from "Your wallets" (self-send is separately blocked by `review()`'s validation, below). A held spot token whose id can't be resolved is **excluded** (never offered). `review()` gates on a selected token, amount (`> 0`, `≤ selected.available`, token decimals) and a destination that is a valid `parseWalletAddress` **and not the user's own master** (self-send → typed reason). The destination field is the shared `RecipientCombobox`, wired by the colocated `use-send-flow-body.ts` (mirrors the withdraw body). `submit` routes on the sendable token's `kind`: `usd` → `usdSend({ destination, amount })` (segregated USDC), `spot` → `spotSend({ destination, token, amount })` (held spot tokens **and** unified USDC); Send stays on Hyperliquid (no L1 fee, instant), so the success state is an instant ✓ (no arrival track), and records the destination via `recordRecipient` so it surfaces under "Recent" next time. `useHyperliquidSend()` projects to `SendState { isApplicable: masterAddress !== null && tokens.length > 0, isComplete: phase === 'sent' }`. **The HL venue's `send` slot is composed in `app/venues.ts`.** mock-venue omits `send`. Suggestion-building + the recent-recipients store used to live inline in this provider; that logic moved to `account/`'s `useRecipientSuggestions` + `shared/components/recipient-combobox`, shared with Withdraw and `agent-balance/`'s Withdraw.
- `EvmCoreFlowProvider`, `useHyperliquidEvmCore`, `EvmCoreFlow` — the HL implementation of the venue-agnostic `VenueEvmCoreCapability` port (move a spot token between **HyperCore** and **HyperEVM**; mirrors the send slice + a direction toggle). `EvmCoreFlowProvider` owns the **EVM⇄Core state machine** (`form → review → signing → sent | error`, with a `direction: core-to-evm | evm-to-core` field; typed errors `amount-invalid | insufficient-balance | wallet-rejected | deposit-required | rate-limited | network | unknown`). - `services/candles-reader.ts` → `hyperliquid-candles-reader` (`debug` on `history fetch retry`; `warn` on `history fetch failed` or `loadOlder failed` after retries exhausted, plus the per-attempt warns/info lines from `withReconnect` for the live WS). `getHistory(symbol, interval)` returns the cached `Candle[]` (empty before `subscribe` primes it). `subscribe(symbol, interval, onUpdate)` (1) fires `gateway.getCandleSnapshot` for `~500 bars` back **with up to 3 attempts** on `network` / `rate-limited` errors (`400 ms` base delay → `800 ms` exponential; `1.5 s` if rate-limited) and emits the whole window as one `{ kind: 'snapshot', candles }` (consumer applies via bulk `setData`) — this is `fetchAndEmitSnapshot()`, run at initial subscribe **and** re-run on every reconnect via `withReconnect`'s `onResync` (ADR-0041), so a reconnect refetches the full window and bulk-replaces rather than replaying the backlog (kills the post-resume "10×" + the permanent gap), (2) opens HL `candle` WS through `withReconnect` (self-healing on initial failure or `failureSignal` abort; `resyncSignal` from the app-scope liveness coordinator force-reconnects it on tab-resume) and emits `{ kind: 'update' | 'new', candle }` per event (`'new'` when the WS event's `openTime` advances past the cache's last bar; `'update'` otherwise — the strict-ascending `openTime` guard drops stale buffered deltas after a resync snapshot lands). `loadOlder(symbol, interval, beforeOpenTime, count)` fetches `gateway.getCandleSnapshot(coin, interval, beforeOpenTime − count·intervalMs, beforeOpenTime − 1)` through the same 3-attempt retry, filters returned candles strictly `< beforeOpenTime`, prepends them to the cache, and returns `{ candles, reachedStart }` (`reachedStart: true` once the upstream returns a short page — caller stops pan-back fetches). `count` is clamped to `[1, 500]` (HL caps each `info.candleSnapshot` response). HL `T/t/o/c/h/l/v` is projected to the domain `Candle` shape.
- `services/tick-aggregation.ts` — pure helpers for the order-book tick picker: `TickChoice = 'auto' | 0.1 | 0.2 | 0.5 | 1 | 10 | 100`, `inferAutoTick()` (always `0` — `'auto'` = native passthrough, no mark-price ladder), `resolveTick(choice)`, and `bucketLevels(rawLevels, tick, side, maxLevels?)` which for `tick > 0` groups raw L2 levels into price buckets (bids round down, asks round up; spread preserved) and for `tick ≤ 0` returns the native levels sorted best-first and capped (no bucketing).

## Cross-app contract

None — the venue talks to Hyperliquid directly (HTTP info + WebSocket subscriptions). It does not call our backend.

### Builder fee contract (ADR-0024)

The partnership builder address and fee rate live in `hyperliquid.constants.ts` as a single source of truth: `HYPERLIQUID_BUILDER_ADDRESS` (same on mainnet and testnet), `HYPERLIQUID_BUILDER_FEE_TENTHS_OF_BPS = 35`, and the per-network `HYPERLIQUID_BUILDER` record (with `maxFeeRate` derived inline from `feeTenthsOfBps`). Approvals are scoped per `(user, builder, chain)` — the same address still requires independent signatures on mainnet and testnet.

**Order-placement contract (deferred to the order-placement PR):** every order action emitted by the exchange gateway, when `useBuilderFee().status === 'approved'`, MUST carry the builder code `{ b: HYPERLIQUID_BUILDER[network].address, f: HYPERLIQUID_BUILDER[network].feeTenthsOfBps }`. When status is anything else (`'missing'`, `'approving'`, or an error variant), the order action MUST omit the builder code entirely. The order-placement PR pulls this from a single helper rather than re-reading the constants at every callsite.

### Wallet-address contract

The venue receives a `getAddress: () => WalletAddress | null` callback. The factory does not capture the address; the stream calls the callback whenever it (re)subscribes. `app/` threads the value from `useAuth().primaryWalletAddress` via a module-scope holder (`apps/client/src/app/wallet-address-holder.ts`) so the venue does not rebuild on wallet rotation. When the address changes, the composition root's auth-mirror effect writes the new value into the holder and then calls `venue.refreshAddress()` — the venue fans out to `stream.refreshAddress()` and `pull.refreshAddress()`, which tear down the active SDK subscription/poller and re-open with the new address. Without this notification a fresh login leaves `webData2` stuck in `disconnected` (subscribers were registered against a null address); other channels work because they are public and never read the address.

### Agent signing-wallet contract

The venue's `trader` capability signs L1 actions with the **agent wallet** (a private key). That key lives only in `AgentWalletProvider`'s in-memory `agentPrivateKeyRef` (never React state, never logged, never serialized). It is bridged to the non-React venue factory using the **same module-scope holder pattern as the wallet address** — `apps/client/src/app/agent-signing-wallet-holder.ts`. The crucial difference from the address holder: it stores a **getter closure**, not a value. The holder never holds the key. `app/agent-signing-wallet-bridge.tsx` (mounted inside `AgentWalletProvider`) registers `useAgentWallet().getSigningWallet` into the holder via an effect and clears it on unmount; `app/venues.ts` passes the holder's `getCurrentAgentSigningWallet` as the factory's `getAgentWallet`. At signing time the getter resolves the agent `AbstractWallet` lazily (`privateKeyToAccount` via `buildAgentSigningWallet`, only when the agent is `approved`); the wallet is built per call and not cached where it can leak. **The key never enters React state, the holder, the venue object, logs (not even redacted-partial), or any `index.ts` export** — only the `HyperliquidAgentWallet` *type* is exported. When no approved agent is available the getter returns `null`, and `trader.placeOrder` / `cancelOrder` / `modifyOrder` return a typed `rejected` `*OrderError` — they never throw or leak. Runtime readiness (wallet connected / onboarded / funded) is the submit gates' job; the capability is never hidden for it.

## Gotchas

- **Four SDK importers, not one.** `@nktkas/hyperliquid` may only be imported by the four gateway files listed in "Depends on" above. The lint rule (`no-restricted-imports`) is the enforcement; the ignore list in `eslint.config.js` is the canonical list.
- **All four live WS streams self-heal through `@/modules/shared/services/with-reconnect` (ADR-0020).** `@nktkas/hyperliquid` reconnects automatically via `@nktkas/rews` and silently auto-resubscribes (`resubscribe: true` default), but its internal retry budget is finite and `failureSignal` aborts once it gives up — at which point the channel is dead. The `l2Book`, `trades`, and `candle` streams have gone through `with-reconnect` since ADR-0010. `web-data2-stream` joined them in the 2026-05-20 ADR-0010 amendment (users were seeing the account-dock indicator stuck red with no data after a daemon-WS drop). All four now retry with exponential backoff (250ms → 30s, ×2, jittered) on both subscribe-time errors and `failureSignal` aborts, and expose a `ConnectionStatusSource` that emits `reconnecting` (amber) between attempts. `with-reconnect` also calls `dead.unsubscribe()` on `failureSignal` so the gateway's multiplex refcount drops cleanly — otherwise the retry would resubscribe under the same multiplex key and pick up a stale, already-aborted bridge in a tight loop.
- **Symbol convention boundary lives in the readers.** HL coin names are bare (`ETH`, `BTC`, …); the domain uses `-PERP` suffixed symbols (`ETH-PERP`, …). The two mapping helpers — `toDomainPerpSymbol(hlCoin)` and `toHlCoin(domainSymbol)` — live in `hyperliquid.utils.ts` and are applied at the reader boundary: `market-data-reader.ts` projects every HL perp's name into `<COIN>-PERP` when building `Market[]` and re-suffixes projected `OrderbookSnapshot.symbol` / `Trade.symbol`; `candles-reader.ts` strips `-PERP` before calling `gateway.getCandleSnapshot` / `gateway.subscribeCandle` and re-suffixes the projected `Candle.symbol`. The gateway is an SDK pass-through and never sees domain symbols. `subscribeOrderbook` / `subscribeTrades` / `subscribeTicker` resolve the HL coin via `resolveHlCoin(symbol)` → `toHlCoinFromMarket(market)` against the refreshed `Market` cache (Phase 3 WIRE-03 fix), with a `toHlCoin` fallback only for symbols absent from the cache (legacy / pre-refresh).
- **Spot `hlCoin` is the raw HL `universe[i].name`, not the display pair and not `@${index}`.** A spot market's `hlCoin` is the exchange's coin key: `@N` for every pair except canonical `PURR/USDC`, whose `name` is the literal pair string `"PURR/USDC"`. The domain `symbol` is the exchange-native pair `BASE/QUOTE` (e.g. `HYPE/USDC`, see ADR-0016) and is rebuilt from the token table — never from `name`; all SDK calls (candles, trades, orderbook, ticker) must use `hlCoin`. `toHlCoinFromMarket(market)` returns `market.hlCoin` so callers never re-derive it. **Do not reconstruct `hlCoin` as `'@' + universe[i].index`** — that coincides with `name` for `@N` pairs but produces a coin HL rejects for `PURR/USDC` (the bug that left PURR's chart/trades/orderbook empty). See ADR-0017.
- **HIP-3 `hlCoin` is `dex:ASSET`, composed once.** A HIP-3 market's `hlCoin` and `symbol` are both `dex:ASSET` (e.g. `xyz:AAPL`). HL's per-dex `metaAndAssetCtxs({ dex })` universe name is **already dex-qualified** for builder perps, so `projectHip3Dex` prepends `dex.name` only when the name is not already qualified (prepending unconditionally produced the `XYZ:XYZ:XYZ100` triple-stack bug). Two HIP-3 markets for the same asset on different dexes (`xyz:AAPL`, `flx:AAPL`) are distinct and must never be collapsed — dedup keys on the full `hlCoin`. `getPerpDexs()[i] === null` marks the main HL DEX (already covered by `getMetaAndAssetCtxs`) and is skipped at the per-dex fan-out call-site.
- **`maxLeverage` is on `Market` for perp & HIP-3 only.** Sourced from the SDK `universe[i].maxLeverage`; absent for spot (no leverage) and legacy mock markets. Backs the market-row tag.
- **Legacy mock-venue markets may lack `marketType` / `hlCoin`.** These fields are optional on `Market`; absent ⇒ treat as `'perp'`. `toHlCoinFromMarket` falls back to the `-PERP` strip for such markets. HIP-3 markets project `hasCandles: true` — the HL `candleSnapshot` endpoint accepts the dex-prefixed coin (`xyz:XYZ100`), which is exactly the form already stored in `Market.hlCoin`.
- **Concurrent identical subscribes are coalesced in the gateway.** `subscribeL2Book`, `subscribeTradesStream`, and `subscribeCandle` share one SDK sub per `(channel, coin[, interval])` via refcount inside `nktkas-hyperliquid-gateway.ts`. React StrictMode double-mount no longer races the shared `WebSocketTransport`: mount-1's unsubscribe decrements the refcount, mount-2's subscribe keeps the sub alive.
- **Candle backfill retries up to 3×.** `gateway.getCandleSnapshot` is wrapped in a tiny inline retry inside `candles-reader.ts` (`network` and `rate-limited` only, 3 attempts total, `400ms → 800ms` base; `1.5s` for rate-limited). Hyperliquid's HTTP info backend occasionally 500s in 2–3-request bursts; three attempts covers the typical burst. Single call site, so the retry is not yet promoted to a shared helper.
- **SDK returns numerics as strings.** All numeric fields in `WebData2Response` / `PortfolioResponse` / `UserFeesResponse` / `DelegatorSummaryResponse` / `BorrowLendUserStateResponse` are stringified per Hyperliquid's wire schema. Use `parseStringifiedNumber` from `hyperliquid.utils.ts`.
- **Spot pricing is keyed by pair, not by token — and `assetCtxs` join by pair `.index`, not array position.** `webData2.spotAssetCtxs[i].coin` is the *pair name* (e.g. "PURR/USDC", "@107"), not the base-token symbol. Spot balances (`spotState.balances[].coin`) and Earn supply (keyed by token index) are token-keyed. Token-symbol → markPx is resolved via `spotMetaAndAssetCtxs` by `buildSpotPriceIndex`: for each `spotMeta.universe[]` pair, base token = `tokens[0]` → `spotMeta.tokens[].name`, **price = `assetCtxs[pair.index]`** (the `@N` id), **NOT `assetCtxs[i]`** — `universe` is a compact list (delisted pairs removed) while `assetCtxs` is sparse/longer, so array-position joins mis-price almost every token (the USD=0 bug). Only **USDC-quoted** pairs (`tokens[1] === 0`) price the base in USD; a non-USDC pair's `markPx` is not a USD value and must not overwrite it. The pull service caches the resulting index.
- **HTTP-only data needs polling.** `portfolio`, `userFees`, `delegatorSummary`, `spotMetaAndAssetCtxs`, `allPerpMetas`, and `borrowLendUserState` have no websocket channel. `services/hyperliquid-pull.ts` polls them every `PORTFOLIO_POLL_MS` (30s) per active wallet and fans the merged snapshot to readers. Errors keep last known values; there is no UI surface yet.
- **REST reads are rate-limited; the cache decorator is the steady-state + reload mitigation.** HL enforces one IP-wide budget of **1200 weight/min** (WS subs are free); `getUserFees`/`portfolio`/metadata/history reads are weight 20+. The historical 429 ramp came from the fee-schedule + volume-history readers each calling `getUserFees` on **every `webData2` tick** (sub-second) — fixed by routing both through the 30s pull's `userFees` snapshot field (ADR-0022 R1). `withGatewayCache` (composed once in `create-hyperliquid-venue.ts`) then absorbs the cold-reload metadata burst persistently. **Do not re-introduce per-tick `getUserFees` in a reader**, and keep live financial state (`portfolio`/`delegatorSummary`/`borrowLendUserState`) and `candleSnapshot`/history reads **off** the cache allowlist — stale balances on reload are worse than a re-fetch.
- **One WS per venue instance.** `web-data2-stream.ts` ensures exactly one upstream subscription regardless of the number of capability readers.
- **`web-data2-stream.ts` / `all-dexs-clearinghouse-state-stream.ts` invalidate `latest` on address rotation.** `ensureSubscribed()` in both nulls the cached `latest` the moment `activeAddress` genuinely rotates (not on first connect, not on address-missing — those already clear it via `stop()`/the disconnected branch), mirroring `hyperliquid-pull.ts`'s `snapshot = EMPTY_SNAPSHOT` reset on address change. Without this, `subscribe()`'s synchronous replay-on-attach (`if (latest !== null) onUpdate(latest)`) hands a new/rerouted listener the **previous** address's data until the fresh tick for the new address lands — this was the root cause of the spectate-mode enter/exit transition flash (Available-to-Trade and Total Equity briefly showing the spectated account's figures). `acting-account-sources.ts`'s divergence selector reroutes a live subscriber straight onto these streams on every spectate toggle, so the invalidation must happen synchronously inside `ensureSubscribed()`, not in a downstream reader.
- **Hexagonal direction.** `hyperliquid/` never imports `trading/` or `portfolio/`. Lint-enforced.
- **Empty-symbol subscribes will close the shared WebSocketTransport.** HL rejects `subscribe` frames with `coin: ""` as `Invalid subscription` AND closes the underlying WS — taking out every sister subscription on the same transport (the bug behind the "all four streams die together" cascade). Two-layer defence is in place: `market-data-reader.ts` and `candles-reader.ts` refuse the subscribe with a `warn` and a no-op `Unsubscribe` when `resolveHlCoin(symbol) === ''`; the trading components (`useOrderbook`, `useTicker`, `useTrades`, `useChart`) additionally skip the subscribe callback when the symbol is the `SUBSCRIPTION_KEY_NONE` sentinel. Do NOT remove either layer when refactoring — they compose to prevent the cascade.
- **`subscribeActiveSpotAssetCtx` is live; the coin MUST be the HL wire key, not the display symbol.** The per-coin spot-ctx subscription works: the SDK helper sends `{ type: "activeAssetCtx", coin: "@N" }` and HL routes spot coins to the `activeSpotAssetCtx` channel at ~1 Hz (verified live 2026-06-11 against `@142` and `PURR/USDC`). An earlier gotcha here claimed HL rejects every per-coin spot-ctx frame ("Bug B", a deliberate no-op stub prescribing a broadcast `spotAssetCtxs` migration) — that claim was disproven and the stub replaced with a real multiplexed subscription (key `activeSpotAssetCtx|${coin}`, distinct from the perp `activeAssetCtx|${coin}` key). The original May 2026 failure was almost certainly a **display symbol** (`BTC/USDC`) sent instead of the wire key (`@N`, or the literal `PURR/USDC` for the canonical pair) — that still yields an `Invalid subscription` error frame (today without closing the shared `WebSocketTransport`). The reader's `resolveHlCoin`/`guardedHlCoin` seam is the guarantee that only wire keys reach the gateway — keep it. The spot ctx structurally lacks `oraclePx`/`funding`/`openInterest`, so the spot ticker renders Mark + 24h Change + 24h Volume only.

## Out of scope

- Real-time refresh of `feeSchedule` from a server-side tier-change signal.
- Any non-`webData2` subscription channel.
- Per-TWAP slice-fill drill-in for a single TWAP id — `twapSliceFills` exposes the full paged history; filtering to a single TWAP's slices is a UI concern, not a separate capability.
- WebSocket subscriptions for TWAP history (`userTwapHistory`); `twapHistory` is a one-shot HTTP read.
- WebSocket subscriptions for order history (`userHistoricalOrders`); `orderHistory` is a one-shot HTTP read. The SDK endpoint accepts no time-window parameters, so paging is not modelled.
