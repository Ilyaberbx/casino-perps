# agent-balance (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns, or its dependencies on other modules.

## Purpose

Reads the User's **Agent Wallet** USDC balance on **Base** (chainId 8453, USDC is 6-decimals) and renders it as one friendly **Agent Balance** metric via `formatUsd`. Per ADR-0046 (D-1, Base half) + `CONTEXT.md` (**Agent Balance**), this figure is explicitly distinct from **Account Value** and from any Venue/DEX collateral. It is **venue-independent**: never routed through a Venue/`Adapter` capability, never a `Balance` row.

## Public surface

- `useAgentBalance(deps?)` → `AgentBalanceViewModel` (`{ balanceUsd: number | null; display: string; status: AgentBalanceStatus; agentWalletAddress: AgentWalletAddress | null }`) — the module's public Agent Balance reading hook. Owns the disconnected gate, the agent-wallet **ensure** flow, and the Base `USDC.balanceOf` read; returns the raw dollar number (for cross-module reconciliation), the pre-formatted display string (`$0.00` when empty / disconnected), the read lifecycle `status` (`'idle' | 'loading' | 'ready' | 'error'`, type `AgentBalanceStatus`, both exported from `index.ts`), and the resolved `agentWalletAddress` — the Base address it read/created, `null` while disconnected / before the read resolves. **Ensure flow (ADR-0078):** GET `/api/agent-treasury/wallet`; on a **404 not-registered** (the fetcher resolves `null`) it creates a user-owned embedded Agent Wallet on the client (`createAgentWallet`, default `useAuth().createAgentWallet`) and registers it (`registerAgentWallet`), then reads the balance off the resulting address. There is **no `userOwned` flag** — the Agent Wallet is user-owned by construction, so the Account Modal's "Export key" gate is plain `useAuth().exportableAddresses` session membership. The `status` exists because `balanceUsd`/`display` alone cannot distinguish "loading" from "loaded zero": `idle` when disconnected, `loading` while the read is in flight, `ready`/`error` once it resolves (ADR-0064 follow-up — consumers like the AI Sheet's persistent balance show a skeleton on `loading`). Venue-independent (never reads a venue/capability). Collaborators (`reader`, `getAgentWallet`, `createAgentWallet`, `registerAgentWallet`) are injectable via `UseAgentBalanceDeps` so it is unit-testable without viem / Privy / HTTP (the create seam's throw is coerced to a `NetworkError`). `useAgentBalanceTile` is a thin projection over it (`display` + `status`) — one source of truth. Consumed cross-module by `trading/`'s AI Sheet for the persistent Agent Balance (slice 08).
- `AgentBalanceTile` — the dumb metric tile. Renders the Base USDC figure (a `ValueSkeleton` while `status === 'loading'`, so the pre-read `$0.00` never flashes as a real balance; an explicit `Unavailable` — not a misleading `$0.00` — when `status === 'error'`, PRD-0007) plus the wallet-gated `AgentBalanceActions` (Deposit / Withdraw triggers). Marked as an AI/agent surface (ADR-0050, amended 2026-06-27): the tile uses the quiet band recipe (faint `--accent` 6% tint, soft 1px inner top highlight) with a static desaturated `AiMascot` glyph (`shared/components/ai-marker`), so it reads distinct from the calm Volume / Fees tiles without neon bloom, dither, or bezel. All state lives in its colocated `useAgentBalanceTile` hook (private).
- `AgentWalletSurface` — the single Agent Wallet surface `app/` mounts once in `AppShell` (it replaced the direct `AgentBalanceSheet` mount). Now a trivial pass-through that always renders the centred `AgentWalletModal` — the Pro side sheet was removed (#273 follow-up), so both Trading Modes open the same modal. It reads the `AgentBalanceSheetProvider` controller through the modal.
- `AgentWalletModal` (private — reached via `AgentWalletSurface`) — the only Agent Wallet surface: reuses the Manage Funds shell (shared `Modal` + left-rail nav + pane) and hosts the three bodies (Deposit | Withdraw | Signing) as tabs, with **no flow-logic rewrite** — its `useAgentWalletModal` hook composes the colocated `useAgentBalanceSheetContent` (moved here from the removed sheet folder) for the resolved deps and maps each nav tab to the provider's `openDeposit` / `openWithdraw` / `openDelegation`. Kept distinct from "Manage Funds" (the Agent Wallet is never trading collateral); the provider `mode` doubles as the open flag and the active tab. `useAgentBalanceSheetContent` (colocated, private) reads the Agent Wallet address + its Base USDC balance (withdraw cap) and binds the explicit-withdraw authorizer + the delegation-grant port (with the `attachAgentSigner` / `removeAgentSigner` seams) and the recipient suggestions (`account/`'s `useRecipientSuggestions({ selfAddress: null })` — every wallet offered, since withdrawing to your own wallet is the default) from `useAuth()` / `account/`. `AgentBalanceActions` shows the three Pro triggers or a single "Manage Agent Wallet" button in Simple (mode resolved in `useAgentBalanceActions`); all triggers now open this modal. **No header wallet-switcher bar** — the Selected Wallet switcher was removed from this modal; the app-header `QuickWalletSwitcher` is the only switcher mount point.
- `AgentBalanceSheetProvider` — the provider unit owning the sheet open/close controller (`{ mode: 'deposit' | 'withdraw' | 'delegation' | null, openDeposit, openWithdraw, openDelegation, close }`). `app/` mounts it in `AccountSessionRoot` so the tile's triggers and the shell's sheet share one source of truth. Consumer hook `useAgentBalanceSheet` is private (consumed within the module by `AgentBalanceActions` + the sheet content hook).
- `DepositFlow` (+ `DepositFlowDeps`) — the dumb, **receive-only** deposit body (issue #211): the Agent Wallet receive address + QR (via shared `CopyableAddress`). Funding happens by sending USDC on Base to that address; there is **no** in-app fund-from-connected-wallet transfer (removed). Stateless — no smart hook.
- `WithdrawFlow` (+ `WithdrawFlowDeps`) — the dumb withdraw body, **structurally matched to the Hyperliquid withdraw form** (shared `AmountInput` with Max, shared `FlowPercentChips`, an available-to-withdraw line, the shared `RecipientCombobox` (`shared/components/recipient-combobox`) as the destination field — fed `walletSuggestions` / `recentSuggestions` from `account/`'s `useRecipientSuggestions` via `WithdrawFlowDeps`, an irreversible-warning `Callout` + `PixelCheckbox` confirm gate shown once a destination is entered, and a fee-less Min / You-receive summary — a direct Base transfer has no protocol fee). Collects a destination Base address + USDC amount and, once the irreversible box is ticked, triggers the **explicit per-action** authorization; on success it calls `WithdrawFlowDeps.onRecordRecipient` so the destination surfaces under "Recent" next time. A single `editing` step (no separate review); the colocated `WithdrawForm` sub-component owns the layout and all state lives in `useWithdrawFlow` (private).
- `DelegationConsent` (+ `DelegationConsentDeps`) — the dumb consent body for the **scoped, revocable delegation** (issue #205): renders the standing delegation status and, when active, the **granted** scope (recipient / cap / expiry) read-only with a revoke; when not active it lets the User **configure the grant** — a USDC **cap** input (`validateDelegationCap`, min $1, client-side max `DELEGATION_CAP_MAX_USD`) + a **TTL preset chip** (`TtlPresetChips`, 7 / 30 / 90 days, default 30) — and offers a one-time grant of the scope built from those inputs. `DelegationConsentDeps` is now `{ recipient, getStatus, getGrantPort, now? }` (the cap/TTL live in the hook state, not injected as a fixed scope; `now` is injected for deterministic `expiresAt` under test). All state lives in `useDelegationConsent` (private). The sheet hosts it on the `delegation` mode (the tile's `Signing` trigger).
- `createDefaultAgentWithdrawAuthorizer(requestExplicitAuthorization)` — the production factory the sheet content hook injects into the withdraw flow (binding the env-backed Base receipt client + the explicit-auth prompt). Type re-exports: `AgentWalletAddress`, `AgentWithdrawAuthorizer`.
- `createDelegationGrant({ client, agentId, address, attachAgentSigner, removeAgentSigner })` / `resolveDefaultGetDelegationStatus(client, agentId)` — the delegation-grant seam + the env-backed status reader the sheet content hook binds. Both are **per-agent** (ADR-0048 D-3): `agentId` scopes which AI Agent's delegation is granted / revoked / read. The grant seam runs the ADR-0078 **3-step handshake** (prepare → attach → confirm) and is threaded the Agent Wallet `address` + the injected Privy `attachAgentSigner` / `removeAgentSigner` seams (defaults from `useAuth()`). `resolveDefaultGetDelegationStatus` resolves the full `DelegationStatusView` (status + granted cap / expiry) — both `DelegationStatusView` and the bare `DelegationStatus` union are exported from `index.ts` so cross-module consumers (e.g. `trading/`) can type the resolved value instead of assuming a bare string. `resolveMinaraRecipient(env)` — the env-derived recipient (USDC → `VITE_MINARA_X402_RECIPIENT`); the cap + TTL are user-configured in the consent surface, not env-derived. (`resolveDelegationScope` was removed — the scope is now built at grant time from the user inputs via `buildDelegationScope`.)
- `BALANCE_UNAVAILABLE_DISPLAY` (`'Unavailable'`) — the single owned display string shown in place of the Agent Balance figure when the live Base-USDC read failed (PRD-0007), so a failed read is never silently rendered as a real `$0.00`. Consumed in-module by `AgentBalanceTile` and cross-module by `trading/`'s AI Sheet (`SheetAgentBalance`), which dropped its byte-identical duplicate to consume this one.

## Owns

- The Base USDC read path: `services/base-usdc-balance-reader.ts` (`createBaseUsdcBalanceReader` — wraps a narrow viem `readContract` slice into `readUsdcBalance(address): ResultAsync<number, BalanceReadFailed>`, converting the raw 6-decimal `uint256` to a dollar number) and `services/base-public-client.ts` (`createDefaultBaseUsdcBalanceReader` — the env-backed production reader over a viem Base public client). The default accessor returns a **lazily-created module-level singleton** (one viem client shared by all three `useAgentBalance` surfaces + the sheet content hook, not a fresh client per mount), wrapped by `services/dedup-usdc-balance-reader.ts` (`createDedupUsdcBalanceReader`) so concurrent reads of the **same address** coalesce onto one in-flight `eth_call` (the de-dup window is in-flight only — a failure is never retained as a cached success). The `createBaseUsdcBalanceClient` / `createBaseUsdcBalanceReader` factories stay un-memoized; only the convenience default is shared. Both are module-internal (not in `index.ts`); the hook's `deps.reader` injection seam is unchanged. Slice OPT-M1, mirrors the server's `RedisAgentBalanceCache` reasoning on the client.
- `api/get-agent-wallet.ts` — thin wrapper on the shared `apiClient` reading the Agent Wallet from the server treasury route (`GET /api/agent-treasury/wallet`). `getAgentWallet(client)` projects to `AgentWalletInfo` (`{ address }`, ADR-0078 — no `userOwned`) and maps a **404 to `null`** (the typed not-registered signal the balance hook branches on to create + register); `resolveDefaultGetAgentWallet(client)` binds it into the zero-arg `GetAgentWallet` fetcher the balance hook injects in production. `getAgentWalletAddress(client)` is the address-only projection (`… | null`) retained for the deposit/withdraw sheet-content path.
- `api/register-agent-wallet.ts` — thin wrapper on the shared `apiClient` registering a client-created Agent Wallet (`POST /api/agent-treasury/wallet`, body `{ address, walletId }`, ADR-0078 register-not-create). `registerAgentWallet(client, { address, walletId })` → `AgentWalletInfo`. The server verifies ownership of `address` and is idempotent.
- `api/delegation.ts` — thin wrappers on the shared `apiClient` for the server delegation routes, **scoped per agent** (ADR-0048 D-3) and following the ADR-0078 3-step handshake: `getDelegationStatus(client, agentId)` (`GET /api/agent-treasury/delegation?agentId=…` → `{ status, appSignerId, capUsd, expiresAt }`, projected to `DelegationStatusView` — the granted `capUsd` / `expiresAt` are `null` when not-granted), `grantDelegation(client, agentId, scope)` (**prepare** — `POST /api/agent-treasury/delegation`, body `{ agentId, ...scope }` → `{ appSignerId, policyId }`, persists nothing), `confirmDelegation(client, agentId, { policyId, ...scope })` (`POST /api/agent-treasury/delegation/confirm` → `{ status }`), `revokeDelegation(client, agentId)` (`POST /api/agent-treasury/delegation/revoke`, body `{ agentId }` → `{ status }`).
- The delegation-grant seam: `services/delegation-grant.ts` (`createDelegationGrant({ client, agentId, address, attachAgentSigner, removeAgentSigner })`) — `grant` runs PREPARE (`grantDelegation`) → ATTACH the app signer client-side (injected `attachAgentSigner` — `false` ⇒ `signer-rejected`, never an active-but-unattached row) → CONFIRM (`confirmDelegation`); `revoke` reads the standing `appSignerId`, removes the app signer client-side FIRST (injected `removeAgentSigner`) THEN marks the server row revoked. Removal is **idempotent / best-effort**: `removeAgentSigner` resolves `true` when the signer is removed OR a benign Privy throw is swallowed (e.g. the signer is already gone), so the authoritative server row-revoke still runs and a stuck-Active delegation self-heals; it resolves `false` only when the owner **declines** the removal (⇒ non-destructive `signer-rejected`, the row stays active). Every server step → a typed `DelegationGrantError('server', …)`; an attach throw ⇒ `signer-failed` (a remove throw is now swallowed at the seam and no longer dead-ends revoke). The Privy attach/remove halves are injected so this module never imports the Privy SDK directly (asserted by `delegation-consent/__tests__/import-boundary.test.ts`).
- `agent-balance.constants.ts` delegation facts: `DELEGATION_ACTION` (`usdc-transfer-with-authorization`, the only representable action — no free-recipient variant), `DELEGATION_CAP_USD` (`50.00`, the **default** cap the form pre-fills), `DELEGATION_EXPIRY_DAYS` (30, the **default** TTL), `DELEGATION_TTL_PRESET_DAYS` (`[7, 30, 90]`, the TTL chip options), `DELEGATION_CAP_MAX_USD` (1000, a **client-side** cap ceiling — the server enforces no max). `agent-balance.config.ts`: `resolveMinaraRecipient` derives the recipient from `VITE_MINARA_X402_RECIPIENT` (a missing / malformed recipient → `null` → no consent offered). `agent-balance.utils.ts`: `validateDelegationCap` (range-checks the cap `[1, DELEGATION_CAP_MAX_USD]`), `buildDelegationScope(recipient, capUsd, ttlDays, now)` (builds the grant scope from the user inputs), `grantedScopeOf(view, recipient)` (projects an active status view to its granted scope, else `null`), and `formatScopeCopy` (projects a scope to display copy — truncated recipient / USD cap / ISO-8601 expiry).
- `agent-balance.constants.ts` — Base chain facts: `BASE_CHAIN_ID` (8453), `BASE_USDC_ADDRESS`, `USDC_DECIMALS` (6), `PUBLIC_BASE_RPC_URL` fallback, the minimal `ERC20_BALANCE_OF_ABI`, and `EMPTY_BALANCE_DISPLAY` (`$0.00`).
- `agent-balance.config.ts` — env-derived `resolveBaseRpcUrl(env)`: the optional `VITE_BASE_RPC_URL` override (validated http(s)) with the public Base RPC fallback. A misconfigured override never crashes and never swaps in a broken transport (mirrors `resolveArbitrumRpcUrl`).
- The Base USDC withdraw path (issue #211): `services/agent-withdraw-authorizer.ts` (`createAgentWithdrawAuthorizer` — `authorizeAndSend(destination, amount)` issuing `USDC.transfer(destination, amount)` on Base over the **explicit per-action** authorization, never the standing delegated signer). A viem-only generic EVM write built against the narrow `BaseTransferWalletClient` / `BaseReceiptClient` slices in `services/base-usdc-transfer.types.ts` (unit-testable with fakes; no venue SDK). `services/base-transfer-clients.ts` binds the env-backed Base receipt client into the production factory. `mapTransferError` (colocated in the withdraw authorizer) is the viem-error classifier, most-specific cause first via `BaseError.walk()`: `wallet-rejected` (`UserRejectedRequestError`), `wrong-network` (`ChainMismatchError`), `insufficient-gas` (`InsufficientFundsError`), `insufficient-balance` (`ContractFunctionRevertedError`, or a mined receipt with `status: 'reverted'`), `receipt-timeout` (`WaitForTransactionReceiptTimeoutError` — not a failure, the transfer may still land), else the residual `transfer-failed`. Messages are scrubbed via `scrubAddresses` before being stored — this function is the module's address-scrub boundary. **Deposit is receive-only** — funding is a direct on-Base USDC send to the Agent Wallet address, so there is no deposit transfer service.
- The deposit / withdraw pure helpers in `agent-balance.utils.ts`: `parseBaseAddress` (viem `isAddress` narrowing), `validateUsdcAmount` (range-checked parse against an available balance), `toUsdcBaseUnits` (6-decimal scaling), and `mapTransferReason` (transfer-error `kind` → user-facing `AgentTransferErrorReason`, with an `assertNever` exhaustiveness guard). USD read-outs go through the shared `formatUsd` (`shared/utils/format-number`) directly — there is no module-local USD formatter.

## Cross-app contract

The Agent Wallet is a **client-created, user-owned embedded wallet** (ADR-0078). Its **address** comes from the server `agent-treasury` module (`GET /api/agent-treasury/wallet` → `{ userPrivyId, walletId, address }`; **404** when not yet registered), but on a 404 the client mints it with `useAuth().createAgentWallet()` and registers it (`POST /api/agent-treasury/wallet` with `{ address, walletId }`). There is **no `userOwned` flag** — the wallet is user-owned by construction (Privy holds its key), so the Account Modal's "Export key" gate is plain `useAuth().exportableAddresses` session membership. The **balance** is read on-chain from Base, not from the server.

The **delegation** lifecycle is owned by the server `agent-treasury` module, is **per-agent** (ADR-0048 D-3, `(userPrivyId, agentId)`), and follows the ADR-0078 **3-step handshake** so there is never an active-but-unattached DB state:

- `GET /api/agent-treasury/delegation?agentId=…` → `{ status, appSignerId, capUsd, expiresAt }` (`status ∈ not-granted | active | expired | revoked`; `appSignerId` lets a later session `removeSigners`; `capUsd` / `expiresAt` are the persisted **granted** scope, `null` when not-granted — the active card shows these real values, not a recomputed default)
- `POST /api/agent-treasury/delegation` (**prepare**) with body `{ agentId, action, recipient, capUsd, expiresAt }` → `{ appSignerId, policyId, expiresAt }` (creates the scoped policy, persists nothing)
- `POST /api/agent-treasury/delegation/confirm` with body `{ agentId, policyId, action, recipient, capUsd, expiresAt }` → `{ status: 'active' }` (persists the row after the client has attached the signer)
- `POST /api/agent-treasury/delegation/revoke` with body `{ agentId }` → `{ status: 'revoked' }` (signer removal is **client-side**, before this call)

The server is the recipient-allowlist authority per agent — it rejects any `recipient` other than the configured x402 address for that `agentId`. V1 has only Minara live. This module supplies the scope (recipient from `VITE_MINARA_X402_RECIPIENT`; **cap + TTL are user-configured** in the Signing tab and validated client-side — the server only checks cap > 0 and expiry in the future, no upper bound), displays the status + scope, and offers grant / revoke. The scoped signer is attached **client-side** via `useAuth().attachAgentSigner` (Privy `addSigners`) between prepare and confirm, and removed via `useAuth().removeAgentSigner` (Privy `removeSigners`) on revoke; both reach this module only through injected seams (never a direct Privy import).

## Gotchas

- **Venue-independent — enforced by test.** `AgentBalanceTile.test.tsx` scans every source file in the module and asserts it never references `venue-provider` / `useVenue` / `useCapability`. Do not reach for the active Venue here, even for convenience — the Agent Balance is the on-chain Base figure, not DEX collateral.
- The hook injects its collaborators (`reader`, `getAgentWalletAddress`) so it is unit-testable without viem / Privy / HTTP; production fills both from the env-backed defaults. A missing address or a failed read sets `status: 'error'` (the `display` stays `$0.00`) — never a throw; the tile and the AI Sheet render an explicit `Unavailable` on that error rather than the misleading `$0.00` (PRD-0007).
- USDC on Base is 6-decimals: `12_500000n` → `12.5`. Do not assume 18 decimals.
- **No direct Privy import — enforced by test.** `delegation-consent/__tests__/import-boundary.test.ts` scans every source file in the module and asserts none imports the Privy SDK. All Privy calls (`createWallet`, `addSigners`, `removeSigners`) live in `account/`'s `AuthBridge` and reach this module only through the injected `createAgentWallet` / `attachAgentSigner` / `removeAgentSigner` seams (defaults from `useAuth()`). Do not import `@privy-io/react-auth` here, even in a comment (the scan matches the literal string).
- **Tile + withdraw motion is locked by CSS-source tests** (#214 modernize-ui polish, ADR-0042/0043). `agent-balance-tile.module.css` and `withdraw-flow.module.css` source all depth/motion from existing tokens — rounded chrome (`--radius-md` surfaces, `--radius-sm` inputs/buttons), soft depth (`--shadow-soft-sm`/`-md` on tile hover; the `--glow-panel` halo was removed), `--accent-glow` (a 1px accent ring + a bounded cyan halo since ADR-0066, = `--glow-accent-sm`) on input focus + primary-button hover, and `--ease-out-quart` curves. Each surface's `*.motion.test.ts` reads the stylesheet source (CSS isn't computed in jsdom) and asserts a `prefers-reduced-motion` block collapsing every transition (and the tile's hover `transform`). (The deposit body is receive-only — address + QR, no CTA — so it carries no motion contract.) The tile's AI treatment (ADR-0050, amended 2026-06-27) uses the quiet band (faint `--accent` tint + soft 1px inner top highlight, no neon ring or dither) — static, not animated — so it does not affect the motion contract; the tile keeps its `--ease-out-quart` hover transition + reduced-motion collapse. Keep that block in sync when adding motion here.

## Out of scope

- The Minara x402 pay path and the server-side execution of the standing, popup-free delegation (server `agent-treasury` owns those). This module owns only the one-time **client consent flow** that grants it (scope + status + revoke) — not the per-call signing. Withdraw here is the **explicit per-action** path only — it must never reach the delegated signer (asserted by `agent-withdraw-authorizer.test.ts`); the consent flow's grant scope can only ever name USDC → the Minara recipient (no free-recipient variant), so it likewise can never move funds to an arbitrary address.
- Any Venue/DEX balance or Account Value figure — those live in `portfolio/` + the venue modules and flow through the `PortfolioReader` / `BalancesReader` capabilities.
