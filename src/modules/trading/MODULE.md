# trading (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns (tables/routes/providers), or its dependencies on other modules.

## Purpose

The trading **Page** — order entry, charts, orderbook, recent trades, account dock. Talks to the active **Venue** through capability ports (`MarketDataReader`, `Trader`, `CandlesReader`, `PositionsReader`, `OpenOrdersReader`, `FillsReader`, `ConnectionStatusSource`, plus the **`ownAccount`** group — `portfolio` / `balances` / `perpsPositionsSnapshot` / `feeSchedule` keyed to the **Acting Address** — read by `use-order-entry` for Available-to-Trade collateral, the Current-Position info row, and spot-market sizing (spot buy → USDC spot `available`, spot sell → base-token holdings), and by `use-leverage-margin-state` for the leverage/margin-mode seed); never imports a venue module directly. **`use-order-entry` and `use-leverage-margin-state` read the order-flow data through `useOwnCapability('…')`, not `useCapabilityOptional('…')`** — the `ownAccount` group keys the **Acting Address** (always the connected Primary Wallet) so order entry, validation/preview, the max slider, the fee estimate, and the leverage seed reflect the User's own account even while **Spectating** (ADR-0038). The account dock + Portfolio keep reading the viewing capabilities.

## Public surface

> **PRD-0008 §9 deletion pass (casino re-skin).** The AI/Minara suggestion surface, the Pro-mode split, and the orderbook / recent-trades / depth / funding / `TickerStats` surfaces were **removed** from this module. Deleted public exports: `PerpSuggestionSheetProvider`, `SuggestionPreviewProvider`, `SuggestionInboxProvider`, `resolveDefaultGetSuggestionInbox`. Deleted feature folders: `perp-suggestion-sheet/`, `suggestion-preview/`, the three suggestion `providers/*`, `orderbook/`, `trades-tape/`, `book-trades-panel/`, `mobile-simple-book/`, `mobile-tab-panel/`, plus `top-bar/TickerStats` + `MobileTickerStrip` and the `api/*-suggestion*` + `services/suggestion-*` files. `TradingPage` is now a minimal compiling shell (TopBar + chart + order entry + equity + account dock); the **Casino-Mode trade screen is rebuilt in a later phase** — prose below that still describes the removed surfaces is stale until then. Everything behaves as the former `simple` density (there is no more trading-mode provider).

- `TradingPage` — the route-addressable page (assembled by `app/`). Desktop renders a two-column shell (chart card + account dock on the left, order entry + equity on the right); mobile renders a compact chart, the equity card, the account dock, a header **Order** button that opens the order ticket in a bottom `Sheet`, and the `MobileTradeDock` footer.
- `HotMarketsTicker` — the global-header "hot markets" running row (a seamless CSS marquee of the selected venue's top markets by 24h volume). Mounted by `app/AppShell` as a full-width band under the nav (hidden on mobile `/trade` and `/portfolio`, where the full header is dropped — `AppShell` instead hands the venue switcher + spectate launcher to the page via `Outlet` context (`AppShellOutletContext.mobileHeaderControls`) and the page renders them in its own header row; gated by the `isMobileChrome` flag). **Route-independent by design:** it reads the venue market universe via the `marketData` capability (the same `useSyncExternalStore(subscribeMarkets, getMarkets)` wiring the selected-market provider uses) and selects via the **URL contract** (`useNavigate` → `/trade?market=hl:<symbol>`, built by `buildTradeMarketHref`), **never** consuming `SelectedMarketProvider` (which only exists inside the `/trade` route). Loading state (`listMarkets()` still empty) renders a shimmer-pill skeleton; "hot" = top-N by 24h volume via `pickHotMarkets` (reuses the shared `filterByMinVolume` floor, with a top-N-of-all fallback so the strip never renders empty on the mock venue). The smart hook + dumb sub-components (`HotMarketItem`, `HotMarketsTickerSkeleton`) live in `components/hot-markets-ticker/`; only `HotMarketsTicker` is re-exported from `trading/index.ts` for `app/` to mount.
- `MobileTradeDock` — the mobile footer composite: the fixed bottom-nav (`MobileBottomNav` — Trade / Portfolio route cells + Account / **Settings** action cells). `use-mobile-trade-dock` drives **Account** via `account/`'s `useIsWalletConnected` + `useAccountModal` / `useAuth().openConnectModal`, and drives **Settings** via `useSettings().open('appearance')` from `shared/providers/settings-provider` — the mobile entry point for the Settings modal. **Place Order is not here:** on mobile the order ticket opens from a compact **Order** button rendered in `TopBar`'s mobile identity row (passed in via `TopBar`'s `mobileAction` prop), which toggles a bottom `Sheet` wrapping the `OrderEntry` form — the open state is owned by `use-trading-page`. Mounted on mobile by **both** `TradingPage` and `portfolio/`'s `PortfolioPage`. Requires the trade-context providers as ancestors (see `TradeDockProviders`). The footer is `position: fixed`; the host page reserves matching bottom padding.
- `TradeDockProviders` — the context stack `MobileTradeDock`'s actions need (`SelectedMarketProvider` → `OrderIntentProvider` → `LeverageMarginProvider`). The `/trade` route already nests these around `TradingPage` (`app/router.tsx`), so it does **not** wrap the dock again; `PortfolioPage` wraps its dock in this since `/portfolio` has none of them.
- `SelectedMarketProvider` — context for the currently selected market across trading sub-components. The context value (`useSelectedMarketContext()`) is `{ selectedMarket: MarketSymbol; setSelectedMarket; market: Market }`: alongside the persisted symbol string it now exposes the **resolved domain `Market`** (`market`, carrying `hlCoin` / `marketType` / `hasCandles`) for the current symbol. The `Market` is resolved once here from the venue's `listMarkets()`/`subscribeMarkets`; panels read `market.hlCoin` as their subscription key instead of re-deriving it. Only the **symbol string** is persisted to URL/storage — the `Market` object is never persisted.
- `FavoritesProvider` — context for the persisted favorites set; mounts inside `TradingPage`, provides `useFavorites()` to all trading sub-components (TopBar, and Phase 6's MarketSelectionWindow). `useFavorites` itself is trading-internal and not re-exported from `index.ts`.
- `MarketSymbol` (type) — the cross-module-relevant market symbol type.
- `OrderIntentProvider` — the **order-intent bus** for assisted-prefill (issue #213). One-way channel to the order ticket: a `publish`ed `OrderIntent` (a directional prefill — side + limit entry + TP/SL bound to a symbol) is read once by the order ticket via `pending`. `app/router.tsx` mounts it between `SelectedMarketProvider` and `LeverageMarginProvider` (so `use-order-entry` reads the same instance). The consumer hook `useOrderIntent()` is trading-internal (not re-exported from `index.ts`). `pending` is reactive state; the ticket applies each **distinct** intent once via a render-time identity tracker (React 19 idiom — no setState in effects), so a re-render never reverts the user's later edits and an off-market intent is ignored. The user always confirms via Place Order — publishing **never** executes an order. (The former suggestion-sheet publisher of intents was removed in the §9 deletion pass.)

The catalog is then gated against the **venue's liquid market set** (ADR-0064 — the single-source-of-truth fix): the hook subscribes to the venue markets reactively (`useSyncExternalStore` over the `marketData` capability, the SAME wiring the Market Selection window uses) and `filterTokensByVenueLiquidity(tokens, venueMarkets, MIN_MARKET_VOLUME_USD)` drops any **venue-listed** symbol below the shared 500k floor — so a market the window hides as illiquid can never appear in the AI feed. A symbol the venue does NOT list (the non-crypto catalog — stocks/commodities/indices/FX/pre-IPO) passes through as the deliberate **AI-only superset**; with no venue list yet (not streamed / venue switch / `listMarkets()` failure) the full catalog shows unchanged (stability fallback). Icons render via the shared `AssetIcon` + `resolveMarketIconUrl` and are warmed on open via the singleton `iconWarmCache.warmMany(collectIconWarmUrls(...))` — **the same Market Selection icon system, no parallel cache**. 247 symbols → no virtualization), the **Estimate → Execute lifecycle** (idle → loading/error → ready; `execute` only enabled when estimated + sufficient balance + active delegation + connected + **fresh** + **venue-onboarded**), the **venue-onboarding gate (slice 07)** — the execute path consumes the SAME `useIsVenueOnboardingReady()` predicate as the Place-Order submit gate (ADR-0026), keyed via `hyperliquid/`'s agent/builder/deposit re-key to the **Selected Wallet** (`(Selected Wallet × venue)`); `canExecute` includes it, and `onExecute` opens the existing `VenueOnboardingSheet` (`useVenueOnboardingSheet().open()`) instead of placing the order when the Selected Wallet isn't onboarded for the venue — so a suggestion can't execute onto a not-onboarded venue. The **estimate freshness gate** (slice 07) — a ready quote is stamped with a client-side `producedAt` (epoch ms) and the hook owns a 1s `setInterval` (injectable as `createInterval`/`now` for tests) driving a live "updated Ns ago" marker; once the quote is older than `ESTIMATE_GRACE_PERIOD_MS` (10_000, in `perp-suggestion-sheet.constants.ts`) it is **stale** (`isEstimateStale`), which blocks execute and swaps the action button for an explicit, **free** `Re-estimate` (re-estimate refreshes `producedAt` and re-enables execute). Staleness takes precedence over sufficiency — a sufficient-but-stale quote still forces a re-estimate; `formatUpdatedAgo` / `isEstimateStale` are pure helpers in `perp-suggestion-sheet.utils.ts`. The **delegation gate** (`getDelegationStatus` per agent → `active | needs-grant`; `onGrantAccess` opens the agent-balance delegation consent; `onTopUp` opens the deposit sheet), and the **History tab** (`HistoryTab`/`HistoryRow` components, `onReopenHistory` opens the preview). On execute (ADR-0073): a `completed` dedup hit passes the cached `StoredSuggestion` to `SuggestionPreviewProvider.open({ suggestion, readOnly: false })` inline as before; a `pending` accept calls `useSuggestionInbox().watch(suggestionId)` and switches the `SuggestTab` action area to the **`SuggestPendingNotice`** working beat ("Generating your suggestion — up to ~90s, you can close this") with the sheet immediately closable (`execute.phase === 'pending'`). If the watched id completes while the sheet is still open, the hook re-reads history on the provider's `historyDirtyVersion` bump and opens the matching row inline (the provider still toasts). History re-opens pass `readOnly: isExpired(row.expiresAt)`. Reaches the engine through `trading/api/estimate-suggestion`, `execute-suggestion`, `get-suggestion-history`, `get-suggestion-inbox`, and `get-suggestion-markets` only — never a venue (the venue market list is read via the `marketData` capability port, and the icon-warm util is reused cross-component from `market-selection-window/`). `ai-agents.constants.ts` declares the `AgentDescriptor[]` registry; `ai-agents.types.ts` holds `AgentId`, `AgentDescriptor`, `AgentParamSchema`. `AgentWorkingLoader` animates the selected agent's icon during the execute loading beat (static poster when `prefersReducedMotion`). The **Suggest stepper** (slice 09) — `SuggestStepper`, a dumb quiet step indicator rendered at the top of `SuggestTab` — communicates the six ordered steps (`DEX → Token → Params → Estimate → Execute → Preview`). The hook owns the derivation: pure `deriveSuggestSteps(flow)` in `perp-suggestion-sheet.utils.ts` projects the flow facts the hook already holds (token chosen → params valid → fresh estimate → can-execute/executing) into per-step `complete | current | upcoming` status, exposed as `steps` on the hook return. - **Suggestion-preview feature folder** (`trading/components/suggestion-preview/`, ADR-0048 slice 10) — the right-side editable preview sheet (`SuggestionPreviewSheet`, `EditableLegs`, `RawSuggestionView`). Internal — not exported from `trading/index.ts`. Smart hook `use-suggestion-preview` seeds the editable legs (`marginUsd | leverage | entry | stopLoss | takeProfit`) from the raw suggestion's `requestParams` + `rawSuggestion`, rebuilds an `OrderDraft` on each edit (with `symbol: suggestion.requestParams.symbol`, so the venue validates the **suggestion's own market**, never the terminal's — ADR-0057), and re-validates via `Trader.validateDraft` (+ `previewOrder`) against **live account state** — this is what catches "had $5 of margin at request, $3 now". Schemas live in `suggestions.schemas.ts`, cross-cutting types in `suggestions.types.ts` (`RoutedSuggestionRequest` — carries a `venueId: SuggestionVenueId` (`hyperliquid | extended`, default `hyperliquid`, slice 04) alongside `agentId` + `params`; `EstimateResult`, `StoredSuggestion`, `RawSuggestion`, `SuggestionMarketsResult`, plus ADR-0073's `SuggestionStatus`, `SuggestionOutcome`, `AcceptSuggestionResult`). Each wrapper exports a `resolveDefault*` factory for production injection.
- **Order-intent provider unit + suggestion→draft mapper** (issue #213). `providers/order-intent-provider/` is the bus (see Public surface). The pure mapper `suggestionToOrderIntent(outcome, symbol, leverage)` (in `trading.utils.ts`, types in `trading.types.ts` — `OrderIntent` / `OrderIntentPatch`) projects a **Directional** `EngineOutcome` into an `OrderIntent`: engine `side` (`long|short`) → order `Side` (`buy|sell`), entry → limit `priceInput`, TP/SL prices → trigger-price inputs (omitted when absent). It returns **null** for Delta-Neutral / No-Trade (nothing to prefill). `leverage` rides on the `OrderIntent` for context but the patch itself carries **no** leverage (applied separately via `LeverageController.setLeverage`).
- **Order-entry feature folder** (`trading/components/order-entry/`) — the `OrderEntry` component tree (internal — not exported from `trading/index.ts`; only `OrderEntry` is re-exported from the folder's own `index.ts`, consumed by `TradingPage`). Its smart hook `use-order-entry` is the **5-way order-type form** (`market | limit | stop-market | stop-limit | twap`, ADR-0034 D-1). It also consumes the **order-intent bus** (`useOrderIntent()`, issue #213): a Directional prefill published by the suggestion sheet is applied once (render-time identity tracker) when the intent's `symbol` matches the active market — seeding side + a limit entry + the TP/SL legs (the protection section auto-enables). The prefill never calls `placeOrder`; spectating still blocks submit and swaps in Stop-Spectating after a prefill. **Validation, parsing, pricing, and sizing are venue-owned (ADR-0035).** The hook assembles a venue-agnostic `OrderDraft` (raw form strings + the leverage selection + the selected-market `symbol` the venue resolves all per-market data from — ADR-0057) and calls `Trader.validateDraft(draft)` (→ the typed `PlaceOrderRequest` fed straight to `placeOrder`, no re-parse, or field-tagged `OrderIssue[]`) and `Trader.previewOrder(draft)` (→ `estimates` for the footer + `capacity.maxCoinSize` for the % slider / MAX). The hook keeps only form state + rendering: it projects the venue issue set into the four inline `isValid` props + the `minOrderValueHint` / `slippageHint` (messages rendered verbatim, mapped by `OrderIssue.field`), owns per-field **dirty/touched** gating (a field's inline error stays hidden until touched; a submit attempt marks all touched), and gates `canSubmit` on the full issue set. `trading/` no longer parses the form, builds the request, converts order sizes, or computes estimates — those dissolved into the venue (`buildRequest` / `coinSizeFor` / `validateForm` and the `size-conversion` / `pre-trade-estimate` / `slippage` / `twap` utils were deleted, along with the venue-fact constants `MIN_ORDER_VALUE_USD` / `DEFAULT_SLIPPAGE_PERCENT` / `MAX_SLIPPAGE_PERCENT` / the TWAP-range). **Entry TP/SL stays trading-owned** (`entry-protection.utils`): the venue-validated request is the base, and `trading/` attaches its `takeProfit` / `stopLoss` legs + the cloid before `placeOrder` (protection size derives from the preview's notional, not a re-implemented conversion). The size-unit toggle keeps a small trading-local coin⇄margin display conversion (a display concern, not a venue order rule). The Available-to-Trade / Current-Position info rows still subscribe `portfolio` / `balances` / `perpsPositionsSnapshot` for display; the venue's `capacity` is the single source for the slider. Live mark is the `useLiveMark` ticker hook (S8); the venue reads its own live-mark cache internally for `previewOrder` (no duplicate subscription). The order-entry test harness builds the venue trader's `validateDraft`/`previewOrder` from `order-entry/__fixtures__/fake-order-validation.ts` (mirrors mock-venue's validation since `trading/` cannot import a venue). The `OrderTypeControl` is a 3-segment row `Market | Limit | [Pro ▾]`: Market/Limit are direct selects, the 3rd segment is a capability-filtered Pro dropdown (`ProTypeMenu` + the colocated `use-order-type-control` listbox state machine) listing only the types the active venue advertises — `stop-market`/`stop-limit` behind `Trader.supportsStopOrders`, `twap` behind `Trader.supportsTwap` (ADR-0034 D-2). With no Pro flag set the 3rd segment is omitted and the control degrades to `Market | Limit`. Per-type fields are conditional sub-components: `StopPriceInput` (stop-market/stop-limit), `PriceInput` with a generalized MID label (limit/stop-limit), `TwapRunningTime` + Randomize (twap). Entry TP/SL (the 2×2 `EntryTpslSection` with coupled price/gain-loss, narrowed on `Trader.supportsTriggerOrders`) is offered for **Market/Limit only** per the UI contract (ADR-0034 D-1). The `PreTradeSummary` footer rows are per-type. **Size units** are `coin | usd` (`SizeUnit`): `coin` is the position size we sign; `usd` is the **margin (collateral)** the user commits, so the leveraged position is `coin = margin × leverage / price` and Order Value = margin × leverage (at 1×/spot, margin == notional). Toggling the unit **converts** the typed value (`convertSizeInput`, coin ⇄ margin) so the underlying order is held fixed; margin is the anchor — on a leverage change the typed margin stays put while the position/order-value re-derive. The `%` slider / MAX sizes off buying power = available margin × `leverage` (from the shared `LeverageMarginProvider`); MAX in `usd` mode commits the full available margin, and a `Margin Required` summary row echoes it. **Minimum order value:** the venue enforces its own notional floor in `validateDraft` (Hyperliquid's `$10`; ADR-0035 D-7 moved the constant into the venue). `trading/` surfaces the resulting size-tagged issue as `minOrderValueHint` / `validation.isOrderValueValid` — reduce-only/closing orders are exempt and the check is skipped when the mark is unknown, both decided by the venue. **Spot markets** (`market.marketType === 'spot'`, exposed as `isSpot`) gate the ticket: Long/Short → Buy/Sell, no `LeverageMargin` (always 1×), no liquidation row, no Reduce-Only, no Margin size unit, and Market/Limit only (the Pro types + TP/SL are suppressed). Spot **sizing is balance-based, not margin × leverage**: a buy is capped by the USDC spot `available`, a sell by base-token holdings (matched on the canonical `baseAsset`; unmatched ⇒ 0, never over-reporting); `availableUnit` flips to `coin` on a spot sell. Spot ticket-gating + the display-only size-unit conversion are still trading-owned; the underlying spot order rules + capacity are venue-owned (`previewOrder`). **HIP-3 markets** (`market.marketType === 'hip3'`, exposed as `isHip3`) stack one more submit gate: `OrderEntry` wraps `SubmitButton` in the shared `<Hip3AbstractionGateButton isHip3>` (inside the ConnectWallet → VenueOnboarding → TradeableFunds stack). For a default (segregated) account on a HIP-3 market it swaps the submit for a one-signature "Enable HIP-3 trading" action (the venue's `hip3Abstraction` capability), which turns on cross-DEX collateral abstraction so the order stops being rejected with "insufficient margin"; a main-dex order or an already-abstracted account passes straight through (ADR-0081). The remaining trading-local helpers live in `entry-protection.utils.ts` / `order-entry.utils.ts`; the order-type constants/types in `order-entry.constants.ts` / `order-entry.types.ts`.

- Internal cross-component hook `use-book-trade-controls` (private) — owns the shared `{tick, sizeAsset, bookSide, baseSymbol, quoteSymbol}` state for the orderbook + trades sub-feature. Consumed by both `BookTradesPanel` (desktop split view) and `MobileTabPanel` (mobile per-tab), so the price-aggregation tick, size denomination, and **order-book side selection** behave identically across layouts. `bookSide` (`'both' | 'bids' | 'asks'`, `BookSide` in `book-trades-panel.types.ts`) drives the nado-style 3-way side picker — the shared dumb `BookSidePicker` (`book-trades-panel/BookSidePicker.tsx`; a `SegmentedControl<BookSide>` of lucide icons with per-option green/red/cyan **soft-tint active states**) rendered at the LEFT of the controls row on the Book tab in **both** `BookTradesPanel` (desktop) and `MobileTabPanel` (mobile, touch-sized); the deliberately control-free Simple-mode `MobileSimpleBook` stays `'both'`. The tick ladder is six options of `market.tickSize × {1,2,5,10,100,1000}` (built by `buildTickLadder` in `orderbook.utils.ts`), scaling correctly across decades (a $200 asset with `tickSize=0.01` yields `0.01,0.02,0.05,0.1,1,10`; a $0.000027 asset with `tickSize=1e-6` yields `0.000001,0.000002,0.000005,0.00001,0.0001,0.001`). ## Gotchas — Trades tape hover-highlight

The Trades tape feature hook (`use-trades`) is the **single owner** of the hover state (`hoveredAddress: WalletAddress | null`) plus the `hoverAddress` / `leaveAddress` / `spectateAddress` handlers — rows never read or own hover state.

`use-trades` also owns `isLoading`, derived from the `TradesUpdate` readiness signal (ADR-0030): `true` until the first `snapshot` lands, then `false` for the session (resets to `true` on market switch via `useAdapterStream`'s `resetOnSubscribe`, not on background reconnect). `TradesTape` and `Orderbook` both render the shared `components/rows-skeleton/RowsSkeleton` (a venue-agnostic shimmer-row placeholder, sized per panel) while their hook reports `isLoading`, so each panel flips once from skeleton to populated instead of building up partially. Once ready with zero trades the tape shows the shared `EmptyState` ("No recent trades") — distinct from the loading skeleton. `TradesTape` forwards `hoveredAddress` down to each `TradeRow`, which forwards it to each `ParticipantCell`. The cell decides per-icon whether its own ghost is highlighted (`address === hoveredAddress` → apply `.spectateButtonMatched`, which recolours the ghost to `--ghost-color-matched` (the `--accent` token) and raises its z-index; the `drop-shadow` glow halo was removed). Highlight is **per-ghost**, not per-row: hovering one ghost recolours every other ghost across the tape whose taker/maker address matches, leaving the row backgrounds untouched. Hover handlers (`onMouseEnter`/`onMouseLeave`/`onFocus`/`onBlur`) live on the ghost `<button>`, not the cell wrapper. Demoable on the mock venue, whose synthetic pool repeats addresses across Taker/Maker roles.

## Cross-app contract

The trading **order/market data** flows through capability ports to a Venue, not the backend. The backend touchpoints are the suggestion endpoints, all Zod-parsed in `trading/api/`:

- `POST /api/suggestions/estimate { agentId, params }` → `{ costUsd, agentBalanceUsd, sufficient }` — no delegation required (quote only). `venueId` (slice 03/04, default `hyperliquid`) scopes the quote to the chosen DEX and is carried **inside `params`** to match the server's `suggestionParamsSchema`; the wrapper (`toRoutedBody`) nests the client's top-level `venueId` there, since a top-level field would be silently dropped.
- `POST /api/suggestions { agentId, params }` → **`202 { suggestionId, status: 'pending' | 'completed', suggestion: StoredSuggestion | null }`** (ADR-0073 D-1, **async durable execute**). The server enqueues a durable BullMQ job and returns immediately: `status: 'pending'` (`suggestion: null`) means the job was accepted — the client registers the id with `SuggestionInboxProvider.watch` and waits for the inbox toast; `status: 'completed'` (a dedup hit within the **`dedupTtlMs` window — 10s**, ADR-0065) returns the cached `StoredSuggestion` for inline render, no waiting. The returned suggestion stays placeable for the longer **`validityTtlMs`/`expiresAt` window — 5 min**. The old synchronous `StoredSuggestion` body is **gone** — slow inference no longer hits this request, so the "NETWORK ERROR / check your connection" banner no longer appears for it (failed *outcomes* surface only via the inbox provider's explicit error toast).
- `GET /api/suggestions/history` → `{ history: StoredSuggestion[] }` — aggregated actor history newest-first, Redis-cached; **server-side filtered to `completed` only** (ADR-0073 D-4). The sheet's History tab refetches on the inbox provider's `historyDirtyVersion` bump and on window focus so a completion that arrived while the tab was closed shows on next open.
- `GET /api/suggestions/inbox` → `{ items: SuggestionOutcome[] }` (ADR-0073 D-5) — the actor's rows from the last ~24h across **all** statuses (`pending | completed | failed`), each `{ id, status, agentId, symbol, style, createdAt, resolvedAt, failureReason }`. The `SuggestionInboxProvider` diffs this to reconcile in-flight work on boot and toast on resolution.
- `GET /api/suggestions/markets?venueId=` → `{ venueId, symbols }` (slice 03/05) — the validate-before-pay token allowlist for the DEX. The sheet gates the static Minara catalog against `symbols` so the client never offers a symbol the server would 422 on; on fetch failure the full catalog is shown and the server remains the final 422 authority (ADR-0062).

See `docs/adr/0001-runtime-venue-switching-via-hard-remount.md` for how venues are swapped, `docs/adr/0008-capability-composed-venues.md` for capability composition, and ADR-0048 for the routed suggestion contract (supersedes ADR-0045).

## Gotchas

- **`trading/` must never import a venue module** (`mock-venue/`, future `hyperliquid/`, …). It depends on capability ports only. Lint-enforced via `import/no-restricted-paths` (see `apps/client/eslint.config.js`).
- The `Venue` is provided by `app/` (composition root), not by trading itself. ADR-0002 places `VenueProvider` in `shared/`, not in trading, so a different consumer (e.g. portfolio) can also read it.
- A venue switch unmounts and remounts the whole trading subtree (hard remount) — keep in-page state stateless w.r.t. venue identity, or accept the remount.
- **MarketSelectionWindow uses `createPortal`** (via the shared `Modal`) for DOM placement — the modal overlay renders at the `document.body` level. However the component remains inside the React tree and therefore has full access to `FavoritesProvider`, `SelectedMarketProvider`, and `VenueProvider` contexts that wrap `TradingPage`.
- **In-ticket dropdowns render through the shared `Popover`** (ADR-0037). The Pro order-type menu (`ProTypeMenu`, driven by `use-order-type-control`) and the TIF / Cross-Isolated `IconSelect` menus would otherwise clip inside the order-entry `.container { overflow:auto }` / the mobile bottom sheet, so each mounts its `role="listbox"` inside `shared/components/popover` while open: `OrderTypeControl` wraps `ProTypeMenu` in `<Popover anchorRef={triggerRef} panelRef={listRef} placement="bottom-end">`, and `IconSelect` does the same for its own list. The listbox state machines (`use-order-type-control`, `use-icon-select`) are unchanged — `Popover` owns placement only; outside-click still works because `listRef` points at the portaled node. The `.proMenu` / `.list` skins keep their border/shadow but no longer set `position`/`top`/`z-index` (the hook writes `position:fixed` inline; stacking is `var(--z-popover)`, which paints above the host sheet/modal per ADR-0079, so the mobile order-sheet's leverage/order-type menus open on top of the sheet — #269). Do **not** revert to a `position:absolute` sibling — it reintroduces the clip.
- **Perp-suggestion sheet motion is locked by a CSS-source test** (#214 modernize-ui polish, ADR-0042/0043; AI marker per ADR-0050). The toggle + sheet header use a **quiet tinted band** (faint `--accent` 6% tint, soft 1px inner top highlight; no neon bloom, dither, or bezel — see ADR-0050 amendment 2026-06-27) with a static desaturated `AiMascot` glyph. The toggle keeps its `--ease-out-quint` idle micro-bob — there is still no `steps(` here. `perp-suggestion-sheet.motion.test.ts` reads the stylesheet source (CSS isn't computed in jsdom) and asserts no `steps(`, a `--ease-out` idle curve, and a `prefers-reduced-motion` block that collapses every loop. Keep that block in sync when adding motion here.
