# shared (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns (tables/routes/providers), or its dependencies on other modules.

## Purpose

Cross-module primitives: the `Venue` **shape** (metadata + per-capability ports), core domain types, design tokens, the `VenueProvider`, and a small set of shared hooks/components. This module is heterogeneous *by design* — it's the place for things that don't belong to a single domain or venue module.

## Public surface

  **Portfolio-tab capability ports** (PRD `0002-portfolio-metrics-full-coverage`, slot ↔ interface; suffix-less slot names per ADR-0008):
  - `openOrdersSnapshot` ↔ `OpenOrdersSnapshotReader` — live snapshot of currently-open orders (`subscribe` only); rows are `Order`, which carries optional `originalSize` / `reduceOnly` / `triggerConditions` (ADR-0023) plus the machine-readable trigger fields `triggerPrice` / `isPositionTpsl` / `triggerKind` (`'tp' | 'sl'`) (ADR-0051) — all projected from the existing HL frontend-orders payload. The trigger fields drive the Positions-table TP/SL read-back column.
  - `twapActiveSnapshot` ↔ `TwapActiveSnapshotReader` — live snapshot of running TWAPs (`subscribe` only); row type `ActiveTwap`.
  - `twapHistory` ↔ `TwapHistoryReader` — completed TWAPs (`subscribe + loadOlder`, one-shot); row type `TwapHistoryEntry`. `TwapHistoryEntry` carries three additive optional fields (ADR-0013 pattern): `durationMinutes?`, `reduceOnly?`, `randomize?` — projected from `record.state` by venues that expose them; absent ⇒ render `--` in the corresponding History columns.
  - `twapController` ↔ `TwapController` (`capabilities/twap-controller.ts`) — write port: `cancelTwap(twap: ActiveTwap): ResultAsync<void, CancelTwapError>` and `cancelAll(twaps: ReadonlyArray<ActiveTwap>): ResultAsync<ReadonlyArray<CancelTwapError>, never>` (collects per-row failures without short-circuiting).   - `twapSliceFills` ↔ `TwapSliceFillsReader` (`capabilities/twap-slice-fills-reader.ts`) — paged reader over TWAP slice fills: `subscribe(onUpdate)` + `loadOlder()`, same structural shape as `TradeHistoryReader` (plugs into `usePaginatedHistoryReader`). Row type is `Fill` (existing domain type). Optional slot on `VenueCapabilities`.
  - `tradeHistory` ↔ `TradeHistoryReader` — user fills history (`subscribe + loadOlder`, paged); rows are `Fill`. `Fill` carries optional `closedPnl` / `direction` / `crossed` / `feeToken` (ADR-0023 + the Side/Type-parity amendment) projected from the existing HL fills payload — absent ⇒ render `--` / fall back to `side`. `crossed` is the taker flag (`true` ⇒ Taker, `false` ⇒ Maker; **`false` ≠ absent** — absent ⇒ `--`); `feeToken` is the token the fee is denominated in (e.g. `'USDC'`), shown verbatim, never converted (absent ⇒ assume `'USDC'`).
  - `fundingHistory` ↔ `FundingHistoryReader` — funding payments (`subscribe + loadOlder`, paged); row type `FundingHistoryEntry`.
  - `orderHistory` ↔ `OrderHistoryReader` — historical orders incl. terminal states (`subscribe + loadOlder`, one-shot); rows are `HistoricalOrder` (status `HistoricalOrderStatus`, plus `HistoricalOrderType` / `HistoricalOrderTif`) — re-exported from `shared/domain`.
  - `interestHistory` ↔ `InterestHistoryReader` — borrow/lend interest entries (`subscribe + loadOlder`, paged); row type `InterestHistoryEntry`.
  - `accountActivity` ↔ `AccountActivityReader` — non-funding ledger updates (deposits/withdrawals/transfers; `subscribe + loadOlder`, paged); row type `AccountActivityEntry`.
  - `marginSummary` ↔ `MarginSummaryReader` (`capabilities/margin-summary-reader.ts`) — `subscribe(onUpdate)` of `MarginSummarySnapshot` (Maintenance Margin, Account Leverage, Margin Ratio %, Unrealized PnL, plus the leverage tooltip's `(a)` total cross positions value / `(b)` cross account value). **Segregated-only**: every field is `null` for unified / portfolio-margin accounts except `marginRatioPct` (`0`). Optional slot; powers the trade-equity-card health surface (ADR-0072).

## Gotchas

- **No top-level barrel.** `import { foo } from '@/modules/shared'` is wrong. Use `import { foo } from '@/modules/shared/hooks/use-is-mobile'` or `'@/modules/shared/domain'`. The carve-out is documented in `frontend-architecture.md`.
- **`shared/components/<leaf>/`** may have an `index.ts` if the leaf bundle has multiple files; a single-file leaf (`shared/components/Foo.tsx`) does not.
- **Promotion rule:** a component lives in its module's `components/` until a **second module** imports it. Do not pre-promote. Reverting a wrong promotion is annoying.
- Adding a new capability is a single new file under `shared/domain/capabilities/` plus an optional slot on `VenueCapabilities`. Existing venues are NOT forced to implement it. Widening an existing capability's interface, however, forces every venue that exposes that slot to update — treat that as a breaking change. See ADR-0008.
- **Trading-write contract (venue-agnostic-trading):** `PlaceOrderRequest` is a discriminated union — `MarketOrderRequest { orderType:'market'; slippageTolerance? }` | `LimitOrderRequest { orderType:'limit'; price; timeInForce }` — over a shared base (`symbol`, `side`, `size`, `reduceOnly?`, `takeProfit?`/`stopLoss?: TriggerLeg`, `clientOrderId?`). It carries **no** `leverage` field (leverage is venue account state set via `LeverageController`, PRD decision 12). The `Trader` port also carries an optional `supportsTriggerOrders?: boolean` flag — the venue-agnostic narrowing signal for the **entry-attached TP/SL** UI section (a venue that places orders but cannot attach `takeProfit`/`stopLoss` legs leaves it absent/`false` and the section is not rendered). `placeOrder` resolves to the discriminated `PlaceOrderOutcome` (`resting` | `filled{averagePrice,filledSize}` | `partially-filled{…,remainingSize}`); a venue rejection under a `status:"ok"` envelope maps to a `PlaceOrderError` of kind `'rejected'`, never to an outcome. `OrderIdentifier` stays `string`; the cloid is a separate `clientOrderId?` field (do not redefine `OrderIdentifier`). The old `OrderAcknowledgement` type was removed.
- **Venue-owned validation + preview contract (ADR-0035 D-1/D-5):** `Trader` carries two **mandatory synchronous** methods (mandatory whenever `trader` is present — read-only venues expose no `trader`, so there is no fallback path; the `?` was dropped in the S2 flip): `validateDraft(draft: OrderDraft): Result<PlaceOrderRequest, OrderIssue[]>` (sync neverthrow `Result`, not `ResultAsync`) parses + validates a raw-string draft — on `Ok` the parsed `PlaceOrderRequest` is the single source of truth fed straight to `placeOrder` (no re-parse in `trading/`), on `Err` the full set of field-tagged issues; and `previewOrder(draft: OrderDraft): { estimates: OrderEstimates; capacity: OrderCapacity }` returns live pre-trade estimates + the % slider / MAX capacity. Supporting venue-agnostic types in `domain.types.ts`: `OrderDraft` (raw form strings — `orderType`/`side`/`sizeUnit`/`sizeInput`/`priceInput`/`stopPriceInput`/`slippageInput`/`timeInForce`/`twapHoursInput`/`twapMinutesInput`/`randomize`/`reduceOnly` — plus the UI-owned `leverage: number` and **`symbol: string`** — the draft is self-describing: the venue resolves every per-market datum (mark, asset, position, margin) from `draft.symbol`, never an ambient active ticker; ADR-0057), `OrderType` (`'market'|'limit'|'stop-market'|'stop-limit'|'twap'`) + `SizeUnit` (`'coin'|'usd'`) (the venue-agnostic mirrors of the `trading/` form unions so `OrderDraft` names them without importing `trading/`), `OrderField` (`'size'|'price'|'stopPrice'|'slippage'|'twapDuration'`), `OrderIssue { field?: OrderField; message: string }` (flat — no `severity`/`source`, D-9), `OrderEstimates = LinearOrderEstimates | TwapOrderEstimates`, and `OrderCapacity { maxCoinSize: number }`. All exported from `shared/domain`'s `index.ts`.

## Out of scope

- Anything app-wide composition (`AppShell`, router) — that lives in `app/`.
- Anything domain-specific (trading-only, portfolio-only) — keep it in the owning module until a second module needs it.
- Server-bound HTTP — `apiClient` lives in `account/`, not here. (Identity is part of every request, so the client lives with identity.)
