# mock-venue (client)

> **Scope of this file:** non-obvious context only. If something is obvious from `index.ts` or by reading a few files, do NOT add it here. Update this file in the same PR that changes the module's public surface, the things it owns (tables/routes/providers), or its dependencies on other modules.

## Purpose

A local **Venue** — implements every capability port from `shared/domain/capabilities/` with deterministic in-memory generators (price, candles, orderbook, trades) and an internal matching engine. Lets the rest of the app run end-to-end with no real backend or exchange. See `CONTEXT.md` for `Venue`, ADR-0008 for the capability composition.

**Dev-only.** The venue is registered behind an `import.meta.env.DEV` guard in `app/venues.ts`, so it is dead-code-eliminated and tree-shaken out of production builds — it never reaches prod and is never an end-user-selectable Venue. The default venue is derived from the registry (`VENUES[0]`), so it is `mock` in dev and `hyperliquid` in prod.

**Trading-capability completeness (verified):** the mock declares AND implements all four trading capabilities — `trader` (with `supportsTriggerOrders: true`, `supportsStopOrders: true`, `supportsTwap: true`, plus the optional `modifyOrder` / `cancelOrder`), `leverageController`, `marginModeController`, and `positionProtection` — so every optional order-entry / row-action UI branch is exercisable without a wallet. `positionProtection` is wired to the live in-memory `accountState.positionsBySymbol` (not a stub), so it rejects `no-position` until a fill opens a position and then records the legs; the wired-venue integration is covered in `services/__tests__/create-mock-venue.test.ts` (capability-shape + place→fill→setProtection→clearProtection). Trigger fills for the recorded TP/SL legs — and for the resting stop orders — are still not simulated.

## Public surface

- `createMockVenue(options?: MockVenueOptions): Venue` — factory. Returns a `Venue { metadata: { id: 'mock', label: 'Mock' }, capabilities: { connection, marketData, portfolio, equityExtensions, feeSchedule, balances, accountMode, positions, perpsPositionsSnapshot, openOrders, openOrdersSnapshot, twapActiveSnapshot, twapHistory, twapController, twapSliceFills, tradeHistory, fundingHistory, orderHistory, interestHistory, accountActivity, fills, trader, leverageController, marginModeController, candles, ownAccount } }`. `twapController` is a no-op stub — `cancelTwap`/`cancelAll` always resolve `ok`. `twapSliceFills` serves a fixed synthetic `Fill[]` via the mock paginated history reader (same one-shot/exhausted pattern as `tradeHistory`). The `ownAccount` group (Acting-Address-keyed order-flow readers, ADR-0038) simply **aliases** the mock's own `portfolio` / `balances` / `perpsPositionsSnapshot` / `feeSchedule` readers — the mock has no address/spectate concept, so acting === viewing (D-5); it omits `refreshActingAddress` (no address-bound state). `accountMode` is a static `{ isSegregated: true }` (the mock has no account abstraction modes — always classic segregated Spot/Perp, so Transfer applies, ADR-0033). The `openOrdersSnapshot` stub returns a deterministic 5-row fixture (`MOCK_OPEN_ORDERS_SNAPSHOT`); `twapActiveSnapshot` returns `MOCK_TWAP_ACTIVE_SNAPSHOT` (2 rows); `twapHistory` is a one-shot reader whose first `loadOlder()` resolves with `MOCK_TWAP_HISTORY` (3 rows) and `exhausted: true`, after which subsequent `loadOlder()` calls are no-ops returning `ok({ exhausted: true })`; `tradeHistory` follows the same one-shot stub shape with `MOCK_TRADE_HISTORY` (3 rows); `fundingHistory` follows the same one-shot stub shape with `MOCK_FUNDING_HISTORY` (3 rows mixing positive and negative funding deltas across long/short positions); `orderHistory` follows the same one-shot stub shape with `MOCK_ORDER_HISTORY` (3 rows covering `filled` / `canceled` / `rejected`); `interestHistory` follows the same one-shot stub shape with `MOCK_INTEREST_HISTORY` (3 rows mixing positive net supply interest, negative net borrow interest, and a non-USDC asset); `accountActivity` follows the same one-shot stub shape with `MOCK_ACCOUNT_ACTIVITY` (3 rows covering `deposit` / `accountClassTransfer` / `withdraw` so the discriminated-delta renderer is exercised end-to-end). The only legitimate entry point.
- `MockVenueOptions` (type).
- `intervalMilliseconds` — utility used by candle generation; exported for reuse where needed.

## Owns

- `services/create-mock-venue.ts` — wires generators + matching engine + connection FSM into one capability-composed `Venue`.
- `services/matching-engine.ts` — local order matching.
- `services/mock-order-placement.ts` — `createMockOrderPlacement(deps)` seam: market-fill / limit-rest + the account-state mutation and order/fill/position/trade emission. `deps` shares `accountState`/`bookStateMap` by reference with the factory (testability seam, not a decoupling boundary). Reads leverage via the injected `deps.leverageFor(symbol)` — the domain `PlaceOrderRequest` no longer carries a `leverage` field (PRD venue-agnostic-trading decision 12).
- `trader` capability: declares `supportsTriggerOrders: true` (the entry TP/SL UI section narrows on this flag; the simulator accepts the `takeProfit`/`stopLoss` legs on the request without yet simulating their trigger fills — full simulation is deferred to the mock-completeness task) plus `supportsStopOrders: true` and `supportsTwap: true` (ADR-0034 D-4 — the mock minimally simulates the Pro types so every order-type flow is demoable + testable without live Hyperliquid). `placeOrder` consumes the discriminated `PlaceOrderRequest`: **market** fills immediately and resolves to a `filled` `PlaceOrderOutcome`; **limit** rests and resolves to `resting`; **stop-market / stop-limit** are recorded as resting open orders awaiting their trigger (emitted on the open-orders stream with a `triggerConditions` string `Price ≥/≤ <stopPrice>` and a recorded price = the post-trigger limit `price` for stop-limit, or `stopPrice` for stop-market) and resolve to `resting` — the trigger fill itself is **not** simulated; **twap** is acknowledged as an accepted, running order and resolves to `resting` (no slice-by-slice execution simulated). `clientOrderId` is echoed through on every variant. Validation: non-positive `size` → `invalid-size`; non-positive limit `price` → `invalid-price`; non-positive `stopPrice` → `invalid-price`; non-positive twap `durationMinutes` → `invalid-size`; unknown `symbol` → `unknown-symbol`. `modifyOrder` mutates a resting limit's price/size in place and re-emits it (resolving `resting`; rejects `not-found` / `invalid-size` / `invalid-price`), `cancelOrder` removes the resting limit. All gated behind the synthetic 50–150 ms ack latency. The `trader` also implements the two optional **synchronous** venue-owned validation/preview methods (ADR-0035 D-1) via `services/mock-order-validation.ts` (`createMockOrderValidation(deps)`): `validateDraft(OrderDraft): Result<PlaceOrderRequest, OrderIssue[]>` parses the raw-string draft into a `PlaceOrderRequest` (or returns field-tagged `OrderIssue`s — `size` for empty/below-$10-notional, `price`/`stopPrice` for missing limit/stop prices, `slippage` for an out-of-band/negative percent, `twapDuration` for a duration outside [5m, 24h]; reduce-only orders are exempt from the min-notional check); `previewOrder(OrderDraft)` returns plausible `OrderEstimates` (linear: notional/margin/fee/liquidation; twap: 30s-cadence slicing) + `OrderCapacity { maxCoinSize }`. The validation seam resolves the market from `draft.symbol` (ADR-0057 — the draft is self-describing; the mock no longer stands in the first market), sources mark price from the live book mid (falling back to the static anchor), and prices capacity off a fixed $900 available margin. Venue-fact constants (`MOCK_MIN_ORDER_VALUE_USD`, `MOCK_MAX_SLIPPAGE_PERCENT`, `MOCK_DEFAULT_SLIPPAGE_PERCENT`, the TWAP range, `MOCK_TAKER_RATE`) live in `mock-venue.constants.ts` (ADR-0035 D-7). Nothing in `trading/` consumes these methods yet (S1 is foundation-only).
- `services/mock-order-validation.ts` — `createMockOrderValidation(deps)`: the pure venue-owned `validateDraft` + `previewOrder` parser/estimator (ADR-0035 D-1). `deps` inject the mark-price reader and the available-margin reader (the market comes from `draft.symbol`, ADR-0057 — no active-symbol resolver) so the parser stays a pure function of (draft, venue state) — testable in isolation (`services/__tests__/mock-order-validation.test.ts`).
- `services/mock-history-reader.ts` — `createMockHistoryReader<T>(rows)`: the one-shot `subscribe + loadOlder → exhausted` reader shared by the six history capabilities (twap/trade/funding/order/interest/account-activity).
- `leverageController` + `marginModeController` capabilities (Task 9): `services/mock-leverage-margin.ts` (`createMockLeverageMargin`) holds per-symbol leverage (default 1×, clamped to the market `maxLeverage`) and margin mode (default cross) in two in-memory maps. `setLeverage` rejects `unknown-symbol` / `invalid-leverage`; `setMarginMode` rejects `unknown-symbol`. The order-placement `leverageFor(symbol)` prefers an open position's leverage, then the controller-set value — so a leverage change before a fill flows into the next position.
- `positionProtection` capability (Task 13): `services/mock-position-protection.ts` (`createMockPositionProtection`) records the TP/SL legs for a symbol's open position in an in-memory map so the position-level Edit-TP/SL UI branch is exercisable without a wallet. `setProtection` rejects `unknown-symbol` / `invalid-trigger` (no leg) / `no-position` (no open position — `accountState.positionsBySymbol.has`); `clearProtection` drops the recorded legs. The mock does not yet simulate the trigger fills.
- `mock-venue.utils.ts` — pure helpers extracted from the factory (`applyDiff`, `updateBookState`, `samplePoisson`, `sampleAckLatencyMilliseconds`, `createBookState`, `currentTimestamp`, `delayMilliseconds`).
- `services/account-state.ts`, `services/portfolio-process.ts` — synthetic account + portfolio. The portfolio process honours both `PortfolioWindow` (incl. `'AllTime'`) and `PortfolioAccountScope` (`'all' | 'perps'`), and populates `spotEquity`/`perpsEquity`/`fourteenDayVolume` on every snapshot. `subscribeSnapshot` projects `pnl` / `volume` as **window-keyed `PortfolioWindowValues`** — each window gets a visibly distinct value (per-window multipliers off the base 24H accumulator) so the Period selector is exercised in dev (ADR-0039). `generatePortfolioHistory` (the chart path) is unchanged.
- `services/connection-fsm.ts` — synthetic connect/disconnect lifecycle so the `Adapter` surface is exercised in tests.
- Generators: `price-process.ts`, `ticker-process.ts`, `candle-generator.ts`, `orderbook-generator.ts`, `trade-generator.ts`. `trade-generator.ts` owns a fixed 8-address synthetic participant pool (`SYNTHETIC_ADDRESS_POOL`) and assigns each generated `Trade` a deterministic `takerAddress`/`makerAddress` drawn (distinctly) from it via the seeded RNG, so the same handful of wallets recur across rows — this is what makes the Taker/Maker (and spectate) features demoable on the mock venue with no live connection.
- `mock-venue.constants.ts`, `mock-venue.types.ts`.

## Cross-app contract

None — the mock venue is fully client-local.

## Gotchas

- **`twapHistory` / `tradeHistory` / `fundingHistory` / `orderHistory` / `interestHistory` / `accountActivity` stubs are `exhausted: true` after the first `loadOlder()`.** Both stubs append their fixed row arrays on the first call and return `ok({ exhausted: true })`; every subsequent `loadOlder()` is a no-op returning the same. Not a bug — `tradeHistory` is a paged reader on the real venue, but the mock collapses it to a single deterministic page so E2E flows on every Portfolio tab terminate predictably (PRD D5).
- **A venue module never imports `trading/`.** Lint-enforced.
- The factory is the entry point. Don't import `services/create-mock-venue` directly from outside the module — go through `createMockVenue()`.
- Generators are pure (no React, no module state outside the closure they're created in). Keep them that way; otherwise the matching engine and tests stop being deterministic.
- The composition root (`app/`) chooses *which* venue to instantiate. Adding a new venue = creating a sibling module (e.g. `hyperliquid/`) and editing only `app/` to wire it. Do not add a venue-selection switch inside `trading/`.

## Out of scope

- Real exchange connectivity — that's a sibling venue module (e.g. future `hyperliquid/`).
- Persistence of mock state across reloads (intentional: mock is ephemeral).
- Identity / auth (lives in `account/`).
